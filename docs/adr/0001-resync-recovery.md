# 0001. 손상 JPEG 복구를 자작 비트 디코더 + 바이트 오라클/resync로 수행

- **날짜:** 2026-06-03
- **상태:** Accepted

---

## 배경

ddrescue로 복구한 디스크 이미지에서 카빙한 JPEG들은 엔트로피 스트림에 **바이트 단위 손상**을 가진다. 바이트 단위로 분석한 결과:

- 손상은 두 종류다. (1) **가짜 마커** `FF XX`(`XX` ∉ `00`,`D0–D7`)는 libjpeg의 스캔을 중단시켜 그 뒤를 **회색**으로 채운다. (2) FF 마커를 만들지 않는 일반 바이트 손상은 Huffman 비트 정렬을 **디싱크**시켜 "윤곽은 보이나 깨진" 출력을 만든다.
- 손상 파일들의 엔트로피에는 **0-채움 구멍이 없었다**(데이터는 존재). 즉 회색은 데이터 손실이 아니라 **디코더가 손상 지점에서 멈추거나 어긋난** 결과다.
- 자작 디코더를 표준 클린 JPEG과 대조해 **정확함**을 검증했다(매끄러운 4:2:2에서 평균 Δ≈0.57=반올림). 따라서 디코더가 디싱크하는 지점 = **실제 손상 위치**다. libjpeg는 이를 자체 오류복원으로 그럴듯하게 보이게 만들 뿐 복원하지 못한다.

기존 복구(`carver/diagnosis.py`+`carver/recovery.py`)는 `FF XX`→`FF 00` 중화 후 libjpeg 강제 디코딩이었다. 이는 마커로 인한 회색만 제거할 뿐, 정렬이 어긋난 경우 garbage를 드러냈고 디싱크를 복원하지 못했다.

## 결정

손상 지점을 정확히 짚고 정렬을 복원하는 엔진으로 전면 교체한다.

- **`carver/jpegdecode.py` — 비트 단위 baseline 디코더(numba).** 표준 디코더와 달리 임의의 시작 비트위치+DC예측값에서 재개할 수 있고, per-MCU 비트위치/DC를 기록하며, 무효 Huffman 코드·계수 오버플로·비정상 비트레이트(디싱크)에서 정확히 멈춘다.
- **`carver/resync.py` — 복구 엔진.** 각 디싱크 지점에서:
  1. **바이트 오라클**: 부근 바이트를 치환/삭제/삽입하며 "이후 clean run을 최대화"하는 편집을 채택(단일바이트 손상 복구, 정렬 보존 → 밀림 없음).
  2. **resync-skip**: 단일바이트로 안 되면 재개 비트위치를 넓게 탐색해 다중바이트 손상/구멍을 건너뜀. `db≈0`인 masking(디싱크 스트림을 DC만 리셋해 재디코딩하는 가짜 복구)은 거부한다.
- **plausibility 채점**: 계수 dequant 경계 + **MCU당 비트레이트 상한(평균 4배)**으로 디싱크를 탐지한다. 비트레이트 체크는 "그럴듯하지만 비트를 비정상적으로 많이 소비하는" 잘못된 정렬을 걸러낸다.
- 좌→우로 처리하므로 편집·세그먼트는 항상 현재 지점 이후에만 일어나 이전 비트위치가 안전하다. 복구 가능한 영역만 복구하고, **물리적으로 소실된 영역은 회색으로 남긴다(가짜 채움 금지)**.

`recover.py`는 분류(diagnosis) 없이 모든 파일에 resync 엔진을 적용하고, 회색 비율 전/후·편집 통계를 `report.csv`에 기록한다.

## 대안

| 대안 | 기각 이유 |
|------|----------|
| 기존 `FF XX`→`FF 00` 중화 + libjpeg 강제 디코딩 | 마커 회색만 제거. 정렬 어긋남(디싱크)은 복원 못 하고 garbage 노출. 비-마커 손상은 미탐지. |
| libjpeg 오류복원에 의존 | 오류복원은 깨진 부분을 그럴듯하게 보이게 할 뿐 복원이 아님. restart 마커 없는 이 카메라 JPEG에서 디싱크 후 전체를 회색/garbage로 만듦. |
| 비트 전수 brute force(모든 비트 정렬 시도) | 너무 느림. clean-run 채점 + 국소 탐색으로 충분. |
| 회색 영역을 디싱크 내용으로 무조건 채움 | 반복 source(건물 등)에서 가짜 반복을 생성. masking 거부로 차단. |

## 결과

**실제 영향**
- `carver/diagnosis.py`, `carver/recovery.py`와 그 테스트를 제거하고 `carver/jpegdecode.py`+`carver/resync.py`로 대체. `recover.py`는 resync 엔진만 호출.
- 샘플 10개 기준 8개가 구조 복원(회색 0.1–31%), 2개는 초기 손상 후 데이터가 물리적으로 소실(95–99% 비트 소진, 재동기 지점 0개)되어 상단만 복원되고 나머지는 가짜로 채우지 않아 회색으로 남는다.

**감수한 트레이드오프**
- 복구 결과는 numpy 배열 → JPEG 재인코딩(손실). 원본 비트스트림을 보존하지 않는다.
- **색 캐스트·밝기 밴드(resync 지점 DC 리셋)·이미지 밀림**은 의도적으로 미해결(별도 과제). resync는 구조 복원에 집중한다.
- `numba` 의존(첫 호출 시 JIT 컴파일 지연). 파일당 바이트 오라클 탐색으로 연산량이 있으나 멀티프로세싱으로 상쇄.

**향후 고려사항**
- 공간 연속성 기반 채점을 더하면 밀림 없는 정확 resync가 가능. 색/밝기/밀림 후처리는 후속 ADR 대상.

# 0003. recover 핫패스를 무손실로 최적화 (GPU·품질 trade-off 배제)

- **날짜:** 2026-06-29
- **상태:** Accepted

---

## 배경

`recover.py output/jpeg -j 12`(무제한)는 822개 처리에 약 80분이 걸렸다. 사용자 관찰로 두 현상이 함께 나타났다: ① CPU 총합은 100%인데 작업 관리자에서 각 워커가 **7–8% ↔ 0%를 주기적으로 진동**(12 logical core라 코어당 100%/12 ≈ 8%), ② 메모리 *용량*은 48%(프로세스당 ~230 MB)로 여유.

요구는 **"복구 품질(출력)은 절대 바꾸지 않고 속도만 개선"**이었다. GPU 가속 가능성도 질의에 포함됐다.

단일 프로세스 cProfile(무거운 파일 `0xA147B000`, 무제한)로 병목을 측정했다(상세 과정은 [조사 기록](../investigations/2026-06-29-recover-perf-profiling.md)):

- `np.insert` **131.7초 / 68%**(221,961회) — `_best_edit`의 삽입 후보 탐색이 바이트 값(0~255)마다 `np.insert(buf, p, v)`로 3 MB `buf`를 통째 복사. 한 파일에서 누적 약 660 GB. 복구 연산과 무관한 낭비다.
- `_probe`(decode_probe 네이티브) 37.7초 / 20% — 실제 비트스트림 디코딩.

진동(①)은 메모리 *대역폭* 포화로 판단했다. 용량은 남아도, 12워커가 동시에 3 MB를 초당 수천 회 복사하면 대역폭이 포화돼 코어가 메모리 대기로 stall(0%)한다.

## 결정

**출력 RGB가 비트 단위로 동일하게 유지되는 변경만** 적용한다(검증: sha256). 품질에 영향을 줄 수 있는 최적화는 채택하지 않는다.

1. **`_best_edit` 삽입 루프 (`carver/resync.py`).** `v`만 1바이트 바뀌므로 `np.insert(buf, p, v)` = `buf[:p] + [v] + buf[p:]`에서 위치 `p`당 버퍼를 1회만 만들고(`work = np.insert(buf, p, 0)`) 루프 안에서는 `work[p] = v`로 값만 교체한다. `np.insert` 호출이 22만 → 약 1,144회로 줄지만 만들어지는 배열은 동일하다.
2. **`_recv_extend` (`carver/jpegdecode.py`).** magnitude를 `s`비트씩 1개 루프로 읽던 것을, 4바이트를 모아 `(acc >> (32 - off - s)) & ((1<<s)-1)`로 한 번에 추출한다. JPEG magnitude category는 4비트(`s ≤ 15`)이고 호출부가 `s > 0`을 보장하므로 `off + s ≤ 7 + 15 = 22 < 32`, 4바이트로 충분하다. 부호확장은 동일.

**근거(측정):** 순수 실행 시간(`bench.py`, cProfile 오버헤드 없음) 기준, ①만으로 무거운 파일 187.7→40.2초(4.67배), ②까지 28.5초(누적 6.59배, 단일 프로세스). 실전 `-j 12` 무제한 전수는 80분 → **9분 43초(8.2배)**, 출력 822개 전부 기존과 동일(action·gray·ops). 실전 배수가 단일(6.59배)보다 큰 것은 `np.insert` 제거로 대역폭 포화(워커 stall)가 풀려 12워커가 깨끗이 병렬화됐기 때문이다.

## 대안

| 대안 | 기각 이유 |
|------|----------|
| GPU 가속 | CUDA 미설치(`cuda.is_available()=False`)에 더해, 병목 88%(복사 68% + 디코딩 20%)가 GPU에 부적합. Huffman 디코딩은 비트스트림 순차 의존(다음 비트 위치가 직전 결과에 의존 → 병렬화 불가), `_probe`는 데이터 의존 분기(SIMT 발산), 후보 평가가 소형 커널 수십만 회(런칭 오버헤드). GPU로 옮길 수 있는 IDCT/색변환(`to_rgb`)은 0.39초로 전체의 0.2%. |
| `maxW`(probe 깊이) 축소·후보 가지치기 | `_probe` 호출을 줄여 빨라지나, `_best_edit`이 후보 간 clean-run 길이를 비교해 최선을 고르는 구분력이 떨어져 **복구 품질이 저하**된다. "출력 불변" 요구 위반이라 배제. |
| `_best_edit` 탐색 루프를 njit으로 융합(Python↔numba 경계 627k회 제거) | 변경 후 `_best_edit` 자체 tottime은 0.33초(0.8%)뿐(측정). 융합해도 줄일 것이 없다. cProfile상 decode_probe 실행은 호출자 `_probe`에 귀속돼 크게 보일 뿐, Python 루프 자체는 병목이 아니다. |
| 비트버퍼 누적(refill) 리팩토링 | 효과 불확실(decode가 메모리 바운드여야 이득). 비트리더 전면 재작성이 임의 시작 비트위치·디싱크 탐지(`mcu_bit` 기록)와 얽혀 버그 위험이 크다. 6.59배 이후 수익체감 구간이라 위험 대비 보류. |
| 프로세스 병렬 → numba 스레드 병렬(prange) | `-j 12`는 대역폭 해소 후 8.2배로 이미 잘 스케일. 스레드 전환은 후보 평가가 `buf`를 공유해(치환은 in-place 후 복원) 스레드 충돌 → 후보별 복사 필요, 복잡도 증가, 이득 불확실. |

## 결과

**실제 영향**
- `carver/resync.py`(삽입 루프)·`carver/jpegdecode.py`(`_recv_extend`) 2파일 15줄 변경. **인터페이스·동작·출력 불변** (기존 테스트 54개 통과).
- 단일 프로세스 6.59배, 실전 `-j 12` **80분 → 9분 43초(8.2배)**. CPU 8%↔0% stall 진동 해소(사용자 작업 관리자 확인).

**감수한 트레이드오프**
- 더 큰 가속(비트버퍼 등)은 위험·수익체감으로 미적용. 남은 시간의 ~88%는 `decode_probe` 네이티브 실행이며, 이는 "실제로 해야 하는 디코딩"이라 무손실로는 추가 큰 개선 여지가 작다.
- `time_budget`은 wall-time 기준이므로, 속도가 빨라지면 같은 budget 안에서 더 많이 복구된다(품질이 올라가는 방향). 단 budget이 걸린 배치는 before/after 출력이 같지 않을 수 있어, **무손실 검증은 `--time-budget 0`으로 해야 한다**.

**향후 고려사항**
- 더 빠르게 하려면 (a) 품질 trade-off(`maxW`/가지치기) 수용 또는 (b) 비트버퍼 리팩토링(검증 비용 큼)이 필요하다. 현 "무손실" 제약에서는 여기가 실용적 한계다.

## 조사 과정 (가설 → 예측 → 증거)

### 1단계 — blocky 정량화: 비영 AC 맵과 세그먼트 대조

- **가설 / 세운 이유:** blocky MCU = AC 계수가 거의 없는(DC 지배) 블록. dc-reset 몽타주에서 관찰됐으므로 재동기 세그먼트 내부, 특히 DC=0 리셋 직후에 몰려 있을 것.
- **실험:** 관찰 출처 파일 중 0xA1F57000(resync 15, undec_after 0.015)·0xBA5F6000(resync 24, 0.041)에 recover를 복제 하니스(`recover_with_buf` — rs.recover와 동일 로직 + 최종 buf 반환)로 재실행, 세그먼트·편집 buf·계수 그리드를 npz로 저장. 블록별 비영 AC 수(|dequant|>0.5, DC 제외)를 MCU 평균으로 집계, blocky = frontier 이전 & 비영 AC ≤ 1.0/블록. 세그먼트별·세그먼트 첫 50 MCU별 비율 산출.
- **예측:** blocky가 재동기 세그먼트(1번 이후)에 집중되고, 세그먼트 머리(리셋 직후)에서 최고.
- **증거:**
  - 하니스 신뢰: 두 파일 모두 report.csv와 동일 결과 재현(0xA1F57000 sub 28·ins 4·resync 15·hole 1·undec 0.015 / 0xBA5F6000 sub 26·ins 3·resync 24·hole 1·undec 0.041).
  - 0xA1F57000: 디코드 영역 평균 비영 AC 30.1/블록, blocky 137 MCU(0.3%). 세그먼트별 0.0~0.9%로 산발, 후반 세그먼트 머리(첫 50 MCU)에서 2~6%.
  - 0xBA5F6000: 평균 30.7, blocky 227 MCU(0.5%). **세그 13~16에 농축**(3.0/3.1/1.5/1.2% vs 나머지 0.0~0.4%), 머리 상승 동일.
  - 렌더 육안(결함 패턴): 두 파일 모두 **재동기 세그먼트들이 수평 밀림 밴드**로 렌더되고 세그먼트별 색 캐스트가 교대. blocky는 산발 소수로 육안 지배 결함이 아님.
- **판단:** 부분 지지 — 특정 세그먼트 농축·머리 상승은 확인되나 비중이 0.3~0.5%로 작다. 육안 품질을 지배하는 것은 blocky가 아니라 **밀림 밴드와 색 캐스트**다(→ 5·6단계). blocky 자체의 복구 가능성은 정렬 재탐색으로 검증한다(2단계).

### 2단계 — blocky 재정렬 시도: clean run은 정렬을 판별하지 못함

- **가설 / 세운 이유:** 1단계에서 blocky가 세그먼트 내부에 농축. 이것이 국소 오정렬(잘못된 재동기 지점)이라면, 부근 다른 비트 정렬 중 blocky가 사라지는(정상 AC) 정렬이 존재하고 clean run으로 구별될 것.
- **실험:** 0xBA5F6000 최장 blocky run(MCU 28520~28527, 7 MCU)을 포함한 구간을, 시작 8 MCU 앞(m=28512)의 비트에서 db ∈ [−1024, +1024] 8비트 간격 전 정렬로 `decode_probe`(W=200), run 비교.
- **예측:** 오정렬이라면 현행(db=0)보다 run이 길거나 같으면서 AC가 정상인 정렬이 소수 존재. 실제 콘텐츠라면 db=0이 유일 최장.
- **증거:**
  - **전 정렬이 run=200(상한 포화)** — db=0 포함 상위 후보 전부 동률. clean run에 판별력이 없다.
  - 대상 run의 비트 폭: 14991036~14991231 = 195비트/7 MCU = **27.9 bits/MCU**(파일 평균 553.8의 5%) — EOB 지배 초저엔트로피 구간.
- **판단:** 판별 불가로 기각도 지지도 아님 — **저엔트로피 구간에서는 clean run이 정렬 신호가 못 된다**(어떤 정렬로 읽어도 유효하게 오래 디코드됨). blocky가 실제 평탄 콘텐츠인지 오정렬 잔재인지 구분하려면 콘텐츠 기반 신호(AC 분포·공간 연속성)가 필요하다. 비중(0.3~0.5%)과 판별 신호 부재를 고려하면 **독립 복구 과제로서 우선순위 낮음** — 밀림 보정(6단계)에 부수해 재평가가 낫다.

### 3단계 — 잔여 46건 ops=0 그룹: SOF 치수 손상 가설 기각, 비카메라 가족 확인

- **가설 / 세운 이유:** [사각지대 조사](2026-07-02-recovery-tool-blindspots.md) 잔여 46건 중 ops=0이 17건인데 전부 비표준 해상도(345×459 ×5, 337×337 ×3, 318×425 ×4, 400×346, 480×640 ×2 등) — 소비자 카메라가 내지 않는 크기라 SOF 해상도 필드 손상을 의심.
- **실험:** 해당 그룹 + 1377×1836 그룹(6건) + 대조군(2816×2112 정상 2건)의 bits/MCU(엔트로피 비트 ÷ 주장 MCU 수)와 파일 끝 EOI 존재를 검사. 카메라 정상 범위는 ~500-800.
- **예측:** 치수 손상이라면 bits/MCU가 정상 범위를 크게 벗어남(주장 MCU가 틀리므로).
- **증거:** 그룹 대부분 bits/MCU 429~681로 **정상 범위**(345×459 547, 318×425 600~652, 1377×1836 408~586; 대조군 474·949). 예외 1건: **0xC91B9000(400×346) 2248** — 단 이 파일은 DQT 전 스텝이 1(사실상 무양자화)이라 고비트레이트가 설명됨(DQT 손상 또는 특수 인코더 의심, 관찰만 기록). 부수 관찰: ops=0 17건 중 9건이 **EOI 부재**(파일 끝 FFD9 없음 = 절단), q0(DC 양자화 스텝)는 5 또는 2 계열(카메라 파일 0xBA5F6000의 q0=1과 다름).
- **판단:** 기각 — 비표준 해상도는 진짜 치수다(bits/MCU 정합). 이 파일들은 카메라 사진이 아닌 **별도 출처/인코더 가족**(0xC8~0xC9 디스크 대역, 소형 비표준 해상도, 절단 다수)으로 보인다. ops=0의 원인은 치수가 아니라 다른 것 → 경계/plain 디코드 대조(4단계).

### 4단계 — 경계 err3 재검: "오탐"이 아니라 드리프트 정탐 (선행 조사 결론 수정)

- **가설 / 세운 이유:** 3단계 그룹은 DQT max [36,30,30](평범)인데 plain full 디코드는 e0 완주(0xC9CD0000 638/638), 경계 ON은 MCU 7~499에서 err 3 정지. [사각지대 조사] 8단계는 이를 "경계 미보정 오탐"으로 결론냈다. 그 프레임이 맞다면 plain 디코드의 계수는 전부 정상 범위여야 한다.
- **실험:** ① plain 디코드 계수 그리드의 max|DC|·max|AC| dequant 측정(물리 상한: DC = 8×(평균−128) ∈ [−1024, +1016]). ② 대상을 넓혀 undec_after>0.15 ∪ (ops=0 & hole≥1) 중 plain e0 완주 40건 전수에 소비 비트율(end_bit/nbits)과 max|DC|를 측정(`bound_scan2.py`).
- **예측:** (오탐 프레임이 맞다면) max|DC| ≤ ~1016+q0/2, 위반은 AC 극단값 소수.
- **증거:** 예측 완전 빗나감.
  - 0xC9CD0000 max|DC| **9540**(1400 초과 블록 2801/3828 = 73%), 0xC8069000 5085, 0xC93EC000 7285 — 물리적으로 불가능한 DC.
  - 40건 전수: 소비율 88.8~100.0%(스트림 길이는 이미지와 정합)인데 **전원 max|DC| 2270~18370** — 온전 스트림 0건.
  - 즉 "plain e0 완주"는 온전성 증거가 아니다: Annex-K 허프만 테이블은 코드 공간이 조밀해 **디싱크된 비트열도 무효 코드 없이 계속 '유효하게' 디코드**되고, DC가 물리 범위 밖으로 무작위 보행한다. 경계 err3 정지는 이 드리프트의 올바른 탐지다.
  - 재해석 — 그럼 왜 ops=0인가: 0xC9CD0000은 정지 MCU 7, 잔여 631 ≥ 450으로 총 MCU 임계로는 수락 가능한데도 재동기가 없었다. 원인은 **연속 손상(경계 발동) 간격이 450 미만** — 450 MCU 무손상 클린런이 존재하지 않으면 어떤 후보도 수락되지 않는다. 총 MCU < 450은 이 조건의 특수형일 뿐이다. 선행 조사의 스케일 임계 실험(수락 ~0.55W)이 4/5를 복구한 것과 정합 — 간격이 100~300인 파일은 스케일 수락으로 풀린다.
- **판단:** [사각지대 조사] 결론 3("경계 quant-비례 미보정")을 **수정**한다: 경계는 대부분 정탐이며, 수락 실패의 본질은 임계의 절대성(간격 < 450)이다. 잔여 의심 2건: (a) AC 경계 마진 — 0xC93EC000 max|AC| 6052(=356×q17)로 경계 6000 근접 초과 1계수(미미), (b) 0xCA9AA000(DQT [49,94,94])·0x95AAD000(비다수파 DQT 지문)의 이식 후 MCU 1~2 정지는 **도너 가족 불일치**(그 계열의 진짜 DHT가 Annex-K가 아닐 가능성)로 재해석 — 이식 도너는 DQT 가족별 매칭이 필요하다(추론, 미실험).

### 5단계 — 색 캐스트의 구조: DC 오프셋은 후처리 상수 보정 가능 (신규 레버)

- **가설 / 세운 이유:** 1단계 렌더에서 세그먼트별 색 캐스트가 지배 결함. DC 디코딩은 차분 누적이므로, 초기 DC 예측값을 바꾸면 그 세그먼트의 모든 블록 DC가 **상수만큼 평행이동**할 뿐이다(디코드 수락 여부는 이미 확정된 뒤이므로). 그렇다면 캐스트 보정은 재디코드가 아니라 **디코드 후 계수 그리드에 (세그먼트, 컴포넌트)별 상수 덧셈**으로 충분할 것.
- **실험:** 0xBA5F6000의 세그먼트별 채택 DC(seg_dc)와, 각 세그먼트 첫 3 MCU 행의 렌더 DC 평균(dequant)을 대조해 캐스트 규모를 정량화.
- **예측:** DC=0 리셋 세그먼트들의 렌더 DC 평균이 0에서 크게 벗어나(캐스트), 세그먼트마다 다를 것.
- **증거:** 24개 세그먼트 중 20개가 zero 채택. 첫 3행 평균 dequant DC: seg 4 Cb **−1055.9**, seg 16 Cr **−1068.3**, seg 21 Cb −1008.6, seg 3 Cb −689.5, seg 13 Cr −796.3 등 — 세그먼트마다 다른 대규모 오프셋(= 밴드별 색 캐스트) 확인. 참고로 물리 상한이 ±1016이므로 이 평균들 자체가 렌더의 DC 오류를 포함함을 보인다.
- **판단:** 지지 — 보정 메커니즘은 구성상 성립(DC 차분의 선형성). 남는 문제는 **오프셋 추정**뿐인데, 6단계(밀림)와 결합해 세그먼트 경계 행 연속성으로 추정하는 것이 자연스럽다. 단 위-행 추정을 재동기 "DC 후보"로 쓰는 방식은 실패했다(막다른 길 참조) — 추정은 반드시 디코드 이후 단계에서.

### 6단계 — 밀림(shift)의 구조와 보정 신호: EOI 앵커는 표본에서 부재, 상대 보정은 가능

- **가설 / 세운 이유:** 1단계 렌더의 지배 결함인 수평 밀림은, `_resync_skip`이 재개 비트를 frontier MCU에 배정하지만 **건너뛴 hole이 몇 MCU분인지(k)를 모르므로** 세그먼트마다 오프셋이 누적되는 구조다. 스트림 끝이 EOI에 정확히 닿으면(마지막 세그먼트 끝 = 총 MCU) 전역 앵커가 생겨 k들의 합이 구속된다.
- **실험:** 0xBA5F6000의 최종 frontier에서 소비 비트(mb[frontier]) 대 전체 비트(nbits) 비교.
- **예측:** 데이터가 EOI까지 살아있는 파일이라면 소비율 100% 근처에서 frontier < total(hole들 몫이 남음).
- **증거:** mb[frontier] = 24,685,045 / nbits 24,685,048 (**100.0%**), frontier 44,572/46,464 — 데이터가 이미지 끝 전에 소진(끝 4% 회색). 이 파일에는 EOI 앵커가 없다(표본 1개 한정).
- **판단:** 부분 지지 — 앵커는 파일에 따라 있을 수도 없을 수도 있다(데이터 소진 파일은 부재). 앵커가 없어도 **세그먼트 경계 행 상관으로 상대 오프셋(k)을 추정해 내부 정합을 맞추는 보정**은 가능하며(전역 1자유도 잔존), 육안 품질에는 이것만으로도 효과가 클 것이다. 임팩트 범위: **resync 세그먼트 총 2029개 / 286파일(RECOVERED 557의 51%)**, 편집 총 sub 3143·ins 419·del 2. 참고: 세그 0(비트 0, 정정렬)에서도 max|DC| 1364 > 1016이 관측돼 — 불완전 바이트 편집의 잔존 DC 오차 또는 국소 디싱크 흔적으로 추정 — **DC 물리 범위 검사가 편집 품질 검증에도 쓰일 수 있다**(신규 레버, 미실험).

### 7단계 — 위-행 DC 추정 후보: 실패 (순환 오염)

- **가설 / 세운 이유:** 5단계의 오프셋 추정을 재동기 단계에서 해결할 수도 있지 않은가 — `_resync_skip`의 DC 후보(캐리, zero)에 "위 행 렌더 DC 평균"(공간 예측)을 추가하면 색 캐스트가 원천 차단되고 clean run도 늘 것(백로그 색 캐스트 항목의 원안).
- **실험:** 0xBA5F6000의 24개 재동기 세그먼트 시작점에서 DC 후보 3종 — zero, carry(dcr), above(위 행 col±2 렌더 DC 평균/q0) — 을 같은 비트에서 `decode_probe`(W=450)로 비교.
- **예측:** above가 zero와 비슷한 run + 정확한 DC(캐스트 없음). 최소한 run이 크게 나쁘지 않을 것.
- **증거:** 예측 빗나감 — above는 24개 세그 중 18개가 run ≤ 67, 2개가 355·406, 4개만 450 포화. carry는 전 세그 450 포화, zero는 세그 4·10(run 23·72)을 빼고 450 포화 — 이 두 세그는 엔진도 캐리 DC를 채택했던 곳이라 정합. est 값 자체가 물리 초과(Y 1145 dequant 등)로 DC 경계(1400)에 즉시 걸리는 경우 다수. 원인: (a) 추정 원천인 위 행 렌더가 **밀림+DC 오류를 포함한 미검증 재구성**이라 순환 오염, (b) 렌더 평균에 클리핑·캐스트가 반영되어 물리 밖 값이 나옴. 부수 관찰: carry와 zero가 대부분 세그에서 동시에 450 포화 — clean run은 DC 품질도 판별하지 못한다(2단계의 정렬 판별 실패와 같은 한계).
- **판단:** 기각. DC 오프셋 보정은 재동기 단계가 아니라 **디코드 후 상수 보정 단계**(5단계)로 옮겨야 한다.

## 기각된 가설 / 막다른 길

- **SOF 해상도 필드 손상(비표준 해상도 그룹) → 기각(3단계).** bits/MCU 429~681로 주장 치수와 정합(예외 0xC91B9000 1건, DQT 이상과 결부). 교훈: **해상도가 "카메라답지 않다"는 인상이 아니라 bits/MCU 정합으로 치수 손상을 판별하라.**
- **"plain e0 완주 = 온전 스트림" 가정 → 기각(4단계, 선행 조사의 오류 원인).** 40건 전원 물리 초과 DC 드리프트. 교훈: **Annex-K 테이블은 코드 공간이 조밀해 디싱크 비트열도 장기간 '유효' 디코드된다 — e0는 온전성 증거가 아니며, 소비율(end_bit/nbits) + DC 물리 상한(±1016+q0/2) 초과 여부로 판별하라.**
- **경계 err3 = 오탐(선행 조사 결론 3) → 수정(4단계).** 대부분 드리프트 정탐. 교훈: **탐지기의 참/거짓 판정은 다른 디코더의 통과 여부가 아니라 물리 불변량으로 검증하라.** (plain 디코드도 같은 코드 공간의 산물이라 독립 증거가 아니다.)
- **blocky 재정렬을 clean run으로 탐색 → 판별 불가(2단계).** ±1024비트 전 정렬 run 포화. 교훈: **저엔트로피(EOB 지배) 구간에서 clean run의 정렬 판별력은 0이다** — 정렬·DC 품질 판별에는 콘텐츠 신호(AC 분포·공간 연속성·DC 물리 범위)가 필요하다.
- **위-행 렌더 DC를 재동기 후보로 → 기각(7단계).** 24개 중 18개 run ≤ 67로 즉시 정지. 교훈: **미검증 재구성(밀림·DC 오류 포함 렌더)에서 파라미터를 추정해 디코드에 되먹이지 말라** — 순환 오염. 보정은 디코드 후 단계로.

## 사용한 방법·도구

일회성 스크립트는 세션 스크래치패드에서 실행(리포에 없음). 핵심 재현 요소:

**① recover 복제 하니스(`recover_with_buf`)** — `rs.recover`와 동일 로직에 최종 buf 반환만 추가.
세그먼트·buf·계수 그리드·mb/dcr을 npz 저장. 재현: 본체 `recover()` 코드를 복사하고 마지막에
`return buf, segments, n, mb, dcr, frontier, rate`로 교체(1단계 증거의 통계 일치로 동일성 검증).

**② 비영 AC 맵**:

```python
ac = cy.copy(); ac[:, :, 0, 0] = 0
nz = (np.abs(ac) > 0.5).sum(axis=(2, 3))                       # 블록별 비영 AC 수
nz_mcu = nz.reshape(mcus_y, vs0, mcus_x, hs0).mean(axis=(1, 3)) # MCU 평균
blocky = decoded & (nz_mcu.reshape(-1) <= 1.0)
```

**③ 온전성 판별(`bound_scan2.py`)** — plain e0 완주 파일에 대해:

```python
done, end_bit, err = dec.decode_full()          # 경계 비활성
ratio = end_bit / dec.nbits                     # 소비율(길이 정합)
mx_dc = max(float(np.abs(g[:, :, 0, 0]).max()) for g in (dec.cy, dec.cb, dec.cr))
# 온전 = 0.97 <= ratio <= 1.01 and mx_dc <= 1400 (물리 상한 1016 + 여유)
```

**④ 정렬/DC 후보 스캔** — `jd.decode_probe(buf, nbits, bit, dc0..., start_m, W, ...)`를
db 그리드(정렬) 또는 DC 후보 3종(zero/carry/above)에 대해 호출, run 비교.

**⑤ 캐스트 정량화** — 세그먼트 첫 3 MCU 행의 렌더 DC 평균: `cb[:, :, 0, 0]` 행 슬라이스 평균(dequant).

## 결론

1. **blocky MCU(백로그 항목)의 판정**: 재동기 세그먼트에 농축(최대 3.1%)·머리 상승(2~6%)은 사실이나 전체 비중 0.3~0.5%로 부차적 결함. 대상 구간은 27.9 bits/MCU급 저엔트로피(EOB 지배)로, clean run이 정렬을 판별하지 못해(전 정렬 포화) **현행 신호로는 복구 불가**. 콘텐츠 기반 채점(AC 분포·공간 연속성) 또는 밀림 보정과의 결합 재평가가 전제. 독립 과제 우선순위 낮음.
2. **육안 품질의 지배 결함은 밀림 밴드 + 세그먼트별 색 캐스트**(resync 2029세그/286파일 = RECOVERED 51%). 이 중 **색 캐스트는 DC 차분의 선형성 덕에 디코드 후 (세그먼트×컴포넌트)별 상수 오프셋 덧셈으로 보정 가능**(재디코드 불필요) — 오프셋 추정만 남음(경계 행 연속성, 밀림 보정과 결합). 밀림은 hole의 MCU 수(k) 미지가 원인이며, EOI 앵커(있으면 절대) + 세그 경계 행 상관(상대)으로 추정 가능. 0xBA5F6000은 소비 100.0%로 앵커 부재 사례.
3. **선행 [사각지대 조사] 결론 3 수정**: 경계 err3은 오탐이 아니라 물리 초과 DC 드리프트의 정탐(plain e0 40건 전원 드리프트, 온전 0건). 수락 실패의 본질은 **연속 손상 간격 < 450**(총 MCU < 450은 특수형) — 임계의 잔여-비례 스케일링(선행 조사 실증 4/5)이 이 그룹 전반의 해법이고, 안전장치로 **DC 물리 범위(±1016+q0/2) 채점**을 병행하면 masking 위험 없이 임계를 낮출 수 있다(미실험 제안). DHT 이식 도너는 DQT 가족별 매칭 필요(0xCA9AA000 계열 재해석).
4. **기타 발견**: 0xC8~0xC9 대역의 비표준 해상도 소형 파일들은 별도 인코더 가족으로 추정(비표준 해상도 + q0 5·2 계열 + ops=0 17건 중 9건 EOI 부재 절단). 0xC91B9000은 DQT 전 스텝 1 + 2248 bits/MCU(DQT 손상 또는 무양자화 인코더, 1건). 세그 0(정정렬)에서도 max|DC| 1364 > 1016 — 불완전 바이트 편집의 잔존 오차 검출에 DC 물리 범위 검사 활용 가능(미실험).

후속 라우팅: [백로그](../backlog.md)에 ① 밀림 보정(경계 행 상관 + EOI 앵커), ② 색 캐스트 후처리 상수 보정(기존 항목 갱신 — 방법 단순화), ③ DC 물리 범위 채점(임계 인하 안전장치 + 편집 검증), ④ 경계 항목 수정(정탐 판명) 반영.

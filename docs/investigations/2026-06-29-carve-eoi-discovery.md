# 조사 기록 — 회색 잔존의 주원인 규명: resync 한계 가설에서 carve 가짜 EOI로

- **날짜:** 2026-06-29
- **한 줄:** 복구 후 회색 잔존을 "resync로 더 짜내려" 추적하다, 원인이 recover가 아니라 **carve의 가짜 EOI 조기종료(추출 누락)**임을 규명. 깊은 탐색·rate 강화·통째로 추출 등 여러 막다른 길을 거쳐 "EOI 직후 stuffing 휴리스틱"([ADR 0002](../adr/0002-carve-eoi-validation.md))에 도달.
- **결론 문서:** [ADR 0002 carve EOI 검증](../adr/0002-carve-eoi-validation.md), [보고서 2026-06-29](../reports/2026-06-29-carve-eoi-recovery.md), [investigation 2026-06-29 0xA1F57000](2026-06-29-resync-limit.md)

> 민감 정보 보호: 사진 내용은 묘사하지 않고 식별자(오프셋)·지표(gray·skew·byte%)로만 기술한다.

---

## 증상

`0x91D5D000`(gray 0.777), `0x9F549000`(gray 0.728)이 resync 복구 후에도 회색이 절반 이상 남았다. 출발 질문은 **"깨진 길이가 길어서 더 깊은 탐색이 필요한가?"**였다(resync 엔진을 더 짜내는 방향 가정).

## 조사 과정 (가설 → 증거)

### 1단계 — resync가 어디서·왜 멈추나

- recover 통계: 두 파일 모두 `hole=1`로 break, resync 1–2회, `gray_before` 0.82–0.84(거의 처음부터 손상).
- verbose recover(`diag.py`)로 **frontier**(recover가 디싱크로 멈춘 MCU 위치, 즉 복구가 도달한 지점)를 추적:
  - `0x9F549000`: frontier 14271 MCU(30.7%)에서 hole. resync 진단 `best run=1`(어떤 재개 비트위치·DC로도 1 MCU도 못 감).
  - `0x91D5D000`: frontier 11414(24.6%), `best run=274`(수락 임계 450 미달 → 거부 → hole).
- DC=0 리셋 비교: 9F는 `zeroDC best=1`, 91D는 280 — 9F는 DC 리셋도 무효.

### 2단계 — 데이터가 남았나: skew와 tail

- **skew**(이미지를 얼마나 채웠나[`MCU%`] 대비 데이터를 얼마나 소비했나[`BYTE%`]의 비율 = `BYTE% / MCU%`; 정상이면 1에 가깝고, 데이터를 과소비하면 커진다)를 측정: 두 파일 모두 **처음부터 skew ≈ 3.5x**(frontier에서 MCU 25–31%인데 BYTE 98–100%; 9F는 3.3x, 91D는 4.0x).
- tail 프로파일: frontier 이후 clean byte 수. 9F tail=**12 byte**(소진), 91D tail=**17784 byte**(데이터 남음 — 1단계의 zeroDC best run 280 MCU와 부합).
- "남은 MCU는 70%인데 남은 데이터는 0–2%"라는 역설이 드러났다.

### 3단계 (막다른 길) — 비트 폭식 가설과 rate 강화

- 가설: skew 3.5x = 어긋난 진행(비트 폭식). `rate_bound`(평균의 4배)가 느슨해 못 잡는다.
- 실험(`exp.py`): `rate_mult`를 2.5/2.0/1.5로 낮춤 → **gray 0.999로 붕괴**(frontier 40에서 멈춤).
- 해석 전환: rate를 낮추니 **정상 데이터를 디싱크로 오판**했다. 즉 skew 3.5x는 어긋난 진행이 아니라 **"데이터가 상단 25–30%만 존재하고 그게 정상 비트레이트"**였다(예: 0x91D5D000은 frontier 11414 MCU에 clean 794365 byte = ≈557 bit/MCU; 0x9F549000은 ≈485). 전체 평균(0x91D5D000 기준 140 bit/MCU)이 데이터 없는 회색 MCU(약 35000개)를 분모에 포함해 낮게 왜곡된 것.
- 교차 검증: 91D tail 17784 byte ÷ 557 bit/MCU ≈ 255 MCU ≈ `best run` 280. 일치.
- 이미지 확인: 상단 25–30%만 정상 디코딩(구조 식별 가능, 색 띠·밀림 동반), 나머지 회색. → 데이터 소실(truncation) 의심으로 기울었다.

### 4단계 (전환점) — carve가 일찍 잘랐나

- 파일명 = 디스크 오프셋. 파일 끝 16 byte가 `FF D9`(EOI), 파일 내 `FF D9` 다수.
- 디스크 원본(`usb.img`)에서 **파일 끝 이후** 프로파일(`disk.py`): 엔트로피 ≈7.97 bit/byte, "FF 다음 00/RST" 비율 **97–99%** = JPEG 엔트로피 연속. 다음 `FF D8 FF`(다음 파일)는 +1.9–2.0 MB 뒤.
- 긴 버전(디스크 오프셋 ~ 다음 `FF D8 FF`)으로 복구(`longrecover.py`): gray 0.728→**0.014**, 0.777→**0.144**. 상단만이 아니라 전체 구조 복원.
- **규명:** carve의 `jpeg_end`가 손상으로 stuffing이 깨져 생긴 **가짜 `FF D9`를 EOI로 오인해 ~30%만 추출**했다. 회색은 복구 실패가 아니라 **추출 누락**이었다.

### 5단계 — 휴리스틱 설계와 4단계 검증

판별 원리: EOI 직후가 엔트로피 연속(stuffing 비율이 높음)이면 가짜, 다른 것(다음 파일 헤더·패딩 → stuffing 낮음)이면 진짜. 아래에서 비교하는 두 방식은:

- **stuffing 휴리스틱**: EOI 직후 4 KB의 stuffing 비율만 보고 가짜 EOI를 건너뛰는 가벼운 방법(이 작업이 채택한 것).
- **통째로 추출**: 다음 파일 시그니처까지 데이터를 통째로 가져온 뒤, recover가 전체 MCU를 다 채우면 자연히 멈추는 끝을 쓰는 방법. 무겁지만 "진짜 끝"에 가까운 기준선.

사용자 요청("비교 측정 먼저")에 따라 네 가지를 차례로 확인했다:

1. **휴리스틱이 기준선(통째로)만큼 정확한가**(`compare.py`): 휴리스틱이 stuffing만으로 정한 끝이, 통째로 가져와 얻은 끝과 2 KB 이내로 일치했다. 게다가 휴리스틱은 진짜 EOI 직후의 잉여 패딩을 빼서 오히려 더 정확했다. → 무거운 디코딩 없이 가벼운 휴리스틱으로 충분.
2. **stuffing 임계값을 깔끔히 정할 수 있나**(의심 파일 92개 + 정상 파일 80개의 stuffing 분포, `dist.py`): 가짜와 진짜의 stuffing 분포가 **겹쳤다**(의심 중앙값 0.60, 정상 중앙값 0.19, 둘을 가르는 깨끗한 골짜기 없음). 7개 샘플로 봤던 "깨끗한 경계"는 착시였고, 임계값에 결과가 민감했다(임계 0.2→0.6으로 올리면 연장 대상이 78→49개로 변함).
3. **임계를 잘못 잡으면 위험한가**(정상 파일 중 stuffing≥0.5인 126개를 일부러 가짜로 보고 연장, `checkrisk.py`): 정상 파일을 가짜로 오판해 더 길게 추출해도 **무해**했다 — recover가 전체 MCU를 다 채우면 멈춰 잉여 데이터를 무시하기 때문(검증한 4개 모두 gray 불변). 손익이 **비대칭**이라(가짜를 놓치면 회색이 남는 손해, 정상을 잘못 연장하면 무해) 임계를 낮게(0.3) 잡아 가짜를 최대한 잡기로 했다.
4. **실제 파일에서 효과**(사용자 제공 샘플 10개를 끝까지 복구, `sample_recover.py`): 개선 7 / 동일(원래부터 복구 불가) 3 / **악화 0**.

## 기각된 가설 / 막다른 길

> 가장 가치 있는 부분. 다음 작업이 같은 길을 다시 헤매지 않게.

1. **"깨진 길이가 길어 깊은 탐색 필요"(초기 가설) → 부분 기각.** resync 탐색 깊이가 아니었다. **짧은 추출본만 보면** 데이터 소실(9F, best=1)·수락 임계(91D, best=274)처럼 보였으나, 디스크 대조 결과 **둘 다 carve 추출 누락**이 진짜 원인이었다(긴 버전 복구로 9F 0.014·91D 0.144). 깊은 탐색이 실제로 필요한 영역은 잔여 일부(`0xA1F57000`류, 별도 investigation)뿐이다.
2. **비트 폭식을 rate 강화로 잡기 → 실패.** `rate_mult`를 평균의 2.5배 이하로 낮추면 gray 0.999로 붕괴한다(정상 고비트레이트 구간을 디싱크로 오판). skew 3.5x는 어긋난 진행이 아니라 "데이터가 일부만 존재 + 평균 왜곡"이었다. **누적 평균 rate로는 국소 비트레이트를 분리할 수 없다**(0xA1F57000 investigation에서도 동일 교훈).
3. **통째로 추출(다음 시그니처까지) → 악화.** carve를 "다음 `FF D8`까지" 추출하면 끝이 또 부정확하다(상한이 헐거움). raw를 5 MB·9 MB로 늘리면 gray 0.144→0.191→0.517로 **악화**된다 — 엔트로피를 디코딩용 비트열로 바꾸는 처리(`destuff`)가 잉여 구간의 `FF D9`까지 데이터로 끌어안아 정렬이 더 어긋나기 때문. **"다음 파일 시작"은 끝이 아니라 탐색 상한으로만** 써야 한다.
4. **coverage 디코딩으로 EOI 검증 → 불필요.** 각 `FF D9` 후보까지 실제로 디코딩해 이미지가 충분히 채워지는지(= coverage)로 진짜 EOI를 가리는 방법. 가장 견고하나 carve에 `jpegdecode` 디코더를 결합해야 하고 디코딩 비용이 든다. 오판이 무해(recover 방어)해서 가벼운 stuffing 휴리스틱으로 충분했다.
5. **stuffing 임계 정밀 튜닝 → 무의미.** 92개 분포가 겹쳐 깨끗한 임계가 없다. 윈도우(4 KB)는 통계적으로 충분(튜닝 불필요). 오판 무해성 덕에 낮은 고정값(0.3)이면 된다.
6. **next_sig를 상한으로 직접 사용 → 회귀.** carve가 넘기는 `next_sig`가 **헤더 내 EXIF 썸네일**(SOS보다 앞, 오프셋 +844 등)일 수 있어, 그대로 상한으로 쓰면 메인 이미지를 844 byte로 잘라먹는다. 엔트로피 시작 이후의 다음 진짜 헤더(`_next_header`: `FF D8 FF E0–EF`)를 탐색해야 한다. → [jpeg-markers §3 가짜 EOI / EXIF 썸네일](../reference/jpeg-markers.md).

## 사용한 방법·도구 (scratchpad 프로토타입)

- `diag.py`: verbose recover(step별 frontier·MCU%·BYTE%·skew, hole 지점 resync 진단·tail 프로파일).
- `disk.py`: 디스크 끝 이후 엔트로피·stuffing 프로파일.
- `longrecover.py`: 디스크에서 긴 버전을 직접 복구 — carve 누락 입증.
- `compare.py`(휴리스틱이 정한 끝 vs 통째로 가져온 끝의 일치도) / `dist.py`(92개의 stuffing 분포) / `checkrisk.py`(정상 파일을 오판해 연장했을 때의 무해성).
- `sample_recover.py`: 10개 샘플 end-to-end.
- `exp.py`: `rate_mult` 실험(막다른 길 2).

## 결론

- **회색 잔존의 주원인은 carve의 가짜 EOI 조기종료(추출 누락)**다 — resync 한계가 아니라. 해결은 EOI 직후 stuffing 휴리스틱([ADR 0002](../adr/0002-carve-eoi-validation.md)), 재추출 실증은 [보고서 2026-06-29](../reports/2026-06-29-carve-eoi-recovery.md).
- 잔존 회색은 두 부류로 갈린다: **진짜 데이터 소실**(truncation; budget0 재복구 후 의심 92개 중 동일 66개, 그중 56개가 추출 크기 불변 = 디스크에도 데이터 없음)과 **resync 미복원**(`0xA1F57000`류: DC 캐리·비트 폭식 한계 → [investigation 2026-06-29 0xA1F57000](2026-06-29-resync-limit.md)).
- 운영: 재추출로 파일이 커지면 recover budget 무제한(`--time-budget 0`)이 필수다(기본 90s는 부족 → 악화).

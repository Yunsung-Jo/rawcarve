# 조사 기록 — recover 느림의 주원인은 _best_edit의 np.insert 전체 복사와 _recv_extend 1비트 추출 루프

- **날짜:** 2026-06-29
- **한 줄:** recover.py가 822개에 ~80분(무제한·-j 12), 각 워커 CPU 8%↔0% 진동. cProfile로 `np.insert`(시간의 68%)가 삽입 후보마다 3 MB `buf`를 통째 복사함을 확인 → 무손실 제거(삽입 위치당 1회) + `_recv_extend`의 1비트 루프를 4바이트 일괄 추출로 교체 → 단일 6.59배, 실전 8.2배(80분→9분 43초), 출력 822개 비트 동일.
- **결론 문서:** [ADR 0003 recover 성능 최적화](../adr/0003-recover-perf-optimization.md), [보고서 2026-06-29 recover 속도](../reports/2026-06-29-recover-speedup.md), [spec 0002 recover](../specs/0002-recover.md)

> 시점 스냅샷(랩 노트). 결론에 이르기까지의 가설·실험·증거·막다른 길을 남긴다.

---

## 증상

`recover.py output/jpeg -j 12`(무제한)가 822개에 ~80분. 사용자 관찰: CPU 총합은 100%인데 작업 관리자에서 각 워커 프로세스가 **7–8% ↔ 0%를 주기적으로 진동**, 메모리 용량은 48%(프로세스당 ~230 MB)로 여유. "다른 앱이 느려진다"는 체감과 함께 GPU 가속 가능성 질의.

- 환경: 12 logical core, numba 0.65.1, **CUDA 미설치**(`cuda.is_available()=False`).

## 조사 과정 (가설 → 증거)

### 1단계 — 시간이 어디서 소비되나
- **가설 / 세운 이유:** 어느 함수가 지배하는지 미상. 추측으로 손대기 전에 측정한다.
- **실험:** 무거운 파일(`0xA147B000`, ops 51, 2816×2112) 단일 프로세스 cProfile, `time_budget` 무제한. (`scratchpad/prof.py`)
- **증거:** (cProfile, 계측 오버헤드로 절대 시간은 부풀려진다 — 순수 실행 시간·배수는 4단계부터 `bench.py`로 따로 측정) 총 193초 중 `np.insert` **131.7초(68%, 221,961회 호출)**, `_probe`(decode_probe) 37.7초(20%), `_best_edit` 자체 16.6초.
- **판단:** `np.insert`가 압도적. 코드 확인 → `_best_edit` 삽입 루프가 `v`(0~255)마다 `np.insert(buf, p, v)`로 3 MB `buf` 전체를 복사한다(위치당 256값 × ~22위치 = 최대 5,632회/edit, `_best_edit` 52회 호출 → 실측 221,961회; clean run이 상한에 닿으면 조기 종료라 상한 미만). 누적 약 660 GB 복사로, 복구 연산과 무관한 낭비.

### 2단계 — CPU 8%↔0% 진동의 정체
- **가설 / 세운 이유:** 1단계에서 `np.insert`가 대량 메모리 복사임을 확인. 12 core라 코어당 100%/12 ≈ 8%. 진동은 **메모리 대역폭 포화**로 의심(용량 48%는 충분하나 대역폭은 별개 자원).
- **실험:** 변경 전/후 작업 관리자 재관찰 + 단일 대비 실전 배수 비교.
- **증거:** 변경 전 8%↔0% 진동, 변경 후 진동 소멸. 실전 배수 8.2배 > 단일 6.59배.
- **판단:** 12워커가 동시에 3 MB를 초당 수천 회 복사 → 메모리 대역폭 포화 → 코어가 메모리 대기로 stall(0%)로 추론. (대역폭 자체를 계측하진 않았다. 진동 소멸 + 실전 배수 8.2 > 단일 6.59가 정황 증거다.) 대역폭 해소분이 단일 개선분에 곱해져 실전이 더 빠르다.

### 3단계 — GPU 적용 가능성
- **가설 / 세운 이유:** 사용자 질의에 따라 연산 가속 후보로 GPU 검토.
- **실험:** 워크로드 성격 분석 + CUDA 가용성.
- **증거:** 병목 88%(복사 68% + 디코딩 20%)는 GPU에 못 올린다 — Huffman 디코딩은 비트스트림 순차 의존(다음 비트 위치가 직전 결과에 의존, 병렬화 불가), `_probe`는 데이터 의존 분기(SIMT 발산), 후보 평가가 소형 커널 수십만 회(런칭 오버헤드). GPU 가능한 IDCT/색변환(`to_rgb`)은 0.39초(0.2%).
- **판단:** GPU 부적합. CPU 측 무손실 최적화로 방향.

### 4단계 — np.insert 무손실 제거
- **가설 / 세운 이유:** 삽입 루프에서 `v`만 1바이트 바뀐다. `np.insert(buf, p, v)` = `buf[:p] + [v] + buf[p:]`이므로 위치 `p`당 버퍼를 1회 만들고 값만 교체하면 결과가 동치다.
- **실험:** `work = np.insert(buf, p, 0)` 후 `work[p] = v`. 변경 전 원본 코드로 만든 baseline과 6개 파일 RGB sha256 비교. (`scratchpad/bench.py`)
- **증거:** `np.insert` 22만 → ~1,144회. 무거운 파일 187.7→40.2초(**4.67배**). sha 6개 전부 동일.
- **판단:** 무손실 4.67배. 채택.

### 5단계 — 재프로파일과 _best_edit 융합 가설
- **가설 / 세운 이유:** `np.insert` 제거 후 `_probe`가 남은 최대. `_best_edit`의 Python 루프를 njit으로 융합해 Python↔numba 호출 경계(627k회)를 없애면 빠를 것이라 추측.
- **실험:** 변경 후 동일 파일 재프로파일(cProfile 39초 / 순수 40.2초).
- **증거:** `_probe` 35.3초(90%), **`_best_edit` 자체 tottime 0.33초(0.8%)**.
- **판단:** `_best_edit`의 Python 루프는 0.33초뿐 → 융합은 헛수고. 가설 기각. 병목은 `decode_probe` 네이티브 실행.

### 6단계 — _recv_extend 1비트 루프 최적화
- **가설 / 세운 이유:** `decode_probe` 핫루프 중 `_recv_extend`가 magnitude를 `s`비트 **1개씩 루프로** 읽는다(`jpegdecode.py:172`). `_peek16`은 이미 상수 시간이나 `_recv_extend`는 루프이므로, 한 번에 추출하면 무손실 가속 여지. (효과는 불확실하다고 보고 sha 안전망으로 시도.)
- **실험:** 4바이트를 모아 `(acc >> (32-off-s)) & mask`로 일괄 추출(`off+s ≤ 7+15 = 22 < 32` 보장). sha로 무손실 검증.
- **증거:** 무거운 파일 40.2→28.5초(추가 **1.41배**), 누적 187.7→28.5(6.59배). sha 6개 동일.
- **판단:** 예상과 달리 효과가 컸다(`_recv_extend`가 실제 핫루프였음). 무손실 채택.

### 7단계 — 추가 무손실 한계와 비트버퍼 보류
- **가설 / 세운 이유:** 28.5초 재프로파일에서 `_probe` 88.6% 여전. 남은 건 `_peek16` + dequant. 비트버퍼 누적(refill) 리팩토링이 후보.
- **실험:** cProfile(njit 내부는 분해 불가) + 비트버퍼 방식의 위험 평가.
- **증거:** `_peek16`은 이미 상수 시간(`_recv_extend` 같은 루프 아님). 비트버퍼는 효과 불확실(메모리 바운드 여부 불명)이고, 임의 시작 비트위치·디싱크 탐지(`mcu_bit` 기록)와 얽혀 핫루프 전면 재작성이 필요해 위험.
- **판단:** 국소 무손실 여지 소진. 비트버퍼는 보류(위험·불확실). 단일 6.59배에서 마무리.

### 8단계 — 무손실 전수 검증
- **실험:** 변경 후 전체 822개 무제한·-j 12 → 기존 `jpeg_recovered`의 `report.csv`와 행 단위(action/gray_after/ops/sub/del/ins/resync/hole/image_size) 비교.
- **증거:** 822개 전부 동일. (앞서 90초 budget 배치에선 2개가 달랐으나, 그 2개를 무제한으로 재복구하니 기존과 ops·gray 일치 → 코드가 아닌 budget 잘림이었다.) 기존 단위·통합 테스트 54개도 전부 통과(회귀 없음).
- **판단:** 무손실 확정.

### 9단계 — before 시간 포렌식
- **가설 / 세운 이유:** 변경 전 실전 시간이 데이터로 없다(사용자 "1시간+"만). 재측정(1시간+) 회피를 위해 그 세션 흔적을 먼저 찾는다.
- **실험:** 기존 `jpeg_recovered` 파일 타임스탬프(첫 출력 ~ `report.csv`) + 세션 로그 명령.
- **증거:** 첫 출력 6/28 23:40:49 → `report.csv` 6/29 01:00:20 = **4,771초 ≈ 80분**. 세션 로그에 `--time-budget 0` 사용. 무거운 파일 2개가 무제한 재복구 결과와 일치(budget 90이면 잘렸을 것) → 기존 = 무제한 확정.
- **판단:** before ≈ 80분(무제한·-j 12·변경 전) 확정. 재측정 불필요.

### 10단계 — 실전 after 측정
- **실험:** 변경 후 전체 822개 무제한·-j 12.
- **증거:** **9분 43초(583초)**. before 80분 / after 9.7분 = **8.2배**. 무손실 822개 일치(8단계).
- **판단:** 실전 8.2배(단일 6.59배 + 대역폭 해소).

## 기각된 가설 / 막다른 길

- **`_best_edit`를 njit 융합해 호출 경계(627k회)를 제거한다 → 헛수고.** 변경 후 `_best_edit` 자체 tottime은 0.33초(0.8%)뿐이었다. **교훈:** njit 함수를 호출하는 Python 루프가 병목일 거라 추측하지 말 것 — cProfile에서 njit 실행 시간은 호출자(`_probe`)에 귀속되므로, 그 함수의 *자체* tottime(0.33)과 njit 실행분을 구분해 확인해야 한다.
- **speedtest를 90초 budget으로 돌려 기존(무제한)과 무손실 비교 → 2개 FAIL.** `gray_after`가 0.093→0.403 등으로 달라 회귀로 오인할 뻔했으나, 두 파일 모두 ops가 적었다(잘림). **교훈:** 무손실(결과 동치) 검증은 반드시 `time_budget=무제한(0)`으로 한다 — `time_budget`은 wall-time 기준이라 속도가 바뀌면 같은 시간에 복구량이 달라져 비교 자체가 성립하지 않는다.
- **GPU 가속 → 부적합.** **교훈:** 비트스트림 순차 의존(Huffman) 워크로드는 GPU에 못 올린다. 데이터 의존 분기 + 소형 커널 다발도 SIMT에 부적합하다.
- **비트버퍼 누적 리팩토링 → 미시도(보류).** 효과 불확실(메모리 바운드 여부 불명) + 핫루프 전면 재작성이 디싱크 탐지와 얽혀 위험. **교훈:** `_peek16`처럼 이미 상수 시간인 연산은 `_recv_extend`(루프)만큼의 무손실 이득 여지가 작다 — 최적화는 루프부터 노린다.

## 사용한 방법·도구

- `scratchpad/prof.py`: 무거운 파일 단일 프로세스 cProfile(tottime/cumtime 상위).
- `scratchpad/bench.py`: 변경 전 baseline(RGB sha256 + stats + time) 저장 → 변경 후 verify로 무손실·속도 비교. 검증 세트는 삽입 경로(`ins>0`)를 타는 파일 5개 + 무거운 파일 1개(합 6개) — 삽입 루프 변경의 무손실을 직접 겨냥.
- `recover.py output/jpeg -j 12 --time-budget 0`: 실전 전수 배치(before/after 동일 조건).
- 기존 `report.csv` 행 단위 비교(무손실 전수).
- 파일 타임스탬프 + 세션 로그(`.claude/projects/.../*.jsonl`) 포렌식(before 시간 복원).

## 결론

- recover 느림의 주원인은 복구 연산이 아니라 `_best_edit` 삽입 후보 탐색의 `np.insert` 전체 복사였다(시간 68%, 누적 약 660 GB). 무손실 제거로 4.67배.
- 다음 핫스팟은 `_recv_extend`의 1비트 추출 루프 → 4바이트 일괄 추출로 추가 1.41배. 누적 단일 6.59배.
- 실전 -j 12에서 8.2배(80분 → 9분 43초): 메모리 대역폭 포화로 인한 워커 stall(CPU 8%↔0%)이 `np.insert` 제거로 해소돼 단일 개선분에 곱해졌다.
- 출력은 822개 전부 비트 동일. 결정과 성과는 [ADR 0003](../adr/0003-recover-perf-optimization.md)·[보고서](../reports/2026-06-29-recover-speedup.md)로 연결한다.
- 관련 한계 기록: [DC 리셋과 비트 폭식 한계](2026-06-29-resync-limit.md)(복구 품질 측 과제, 본 조사의 성능 축과 별개).

# 조사 기록 — JPEG 회색·깨짐의 디싱크 복구

- **날짜:** 2026-06-03
- **한 줄:** 카빙된 손상 JPEG의 "회색 영역"과 "깨진 영역"의 원인을 바이트 단위로 규명하고,
  bit-level 디코더 + 바이트 오라클/resync 복구 엔진에 도달하기까지의 조사 과정.
- **결론 문서:** [ADR 0001](../adr/0001-resync-recovery.md)(결정),
  [recover 스펙](../specs/0002-recover.md)(동작),
  [엔트로피·디싱크 원리](../reference/jpeg-entropy-coding.md)(원리).

> 이 문서는 **시점 기록(랩 노트)**이다. 결론에 이르기까지의 가설·실험·증거·막다른 길을 남겨,
> 다음 작업이 같은 길을 다시 헤매지 않고 방법론을 재사용하도록 한다. 살아있는 문서가 아니므로
> 이후 코드가 바뀌어도 갱신하지 않는다(당시의 기록). 표본 파일은 디스크 내 오프셋으로만 지칭한다.

---

## 증상

추출된 JPEG 다수가 **상단 일부만 정상이고 나머지가 통회색**, 또는 **윤곽은 보이나 색·위치가
어긋난 깨짐**을 보였다. 사용자 지정 표본 10개(`0x41D4C000`, `0x17839000`, `0x42243000`,
`0x44CA1000`, `0x49848000`, `0x16F49000`, `0x17643000`, `0x44D9F000`, `0xD13BF000`,
`0xC9401000`)로 조사했다.

## 조사 과정 (가설 → 증거)

### 1. 바이트 구조 분석
- 표본 전부 **baseline(SOF0), RST/DRI 없음**, 4:2:2(8개) 또는 4:2:0(2개).
- 엔트로피에 **0런·상수런·FF런이 없음**(≥64B 기준) → ddrescue의 0-채움 구멍이 아니다. 데이터는 존재한다.
- 스캔 데이터 안에서 가짜 마커 `FF XX`(XX∉{00,D0–D7})를 발견. 절단점 이분탐색으로 잰
  **디코더 정지 위치가 첫 가짜 마커 위치와 일치**(10개 중 9개).
- → 1차 가설: 회색 = 가짜 마커가 libjpeg의 스캔을 중단시켜, 남은 MCU를 중간 회색(계수0 →
  레벨시프트 +128 → RGB 128)으로 채운 것.

### 2. 중화 실험 — 1차 가설의 한계 발견
- 가짜 마커 `FF XX` → `FF 00`으로 중화한 뒤 디코딩.
- 일부 표본은 회색이 사라지고 구조가 복원됐으나, **다수는 회색이 컬러 garbage/단색 밴드로 바뀜**.
- 회색 비율 지표만 보면 "복구된 것처럼" 보이지만(밴드는 무채색이 아니므로), 실제로는 진짜 내용이 아니었다.
- → 깨달음: **회색(디코더 정지)과 깨짐(어긋난 진행)은 같은 "비트스트림 디싱크"의 두 얼굴**이다.
  RST 앵커가 없어 한 번 어긋나면 끝까지 번진다. "회색만 따로 제거"는 불가능 — 디싱크 자체를 고쳐야 한다.

### 3. 자작 bit-level 디코더 + 검증
- libjpeg(PIL)는 손상을 자체 오류복원으로 **은폐**해 정지/어긋남 위치를 가린다 → 임의 비트위치에서
  재개 가능하고 디싱크를 정확히 보고하는 디코더가 필요했다.
- numba로 bit-level baseline 디코더 작성. 표준 클린 JPEG(4:2:2/4:2:0)을 PIL과 대조: 매끄러운
  영상 평균 Δ≈0.57(반올림 수준), 날카로운 엣지·고대비에서도 디싱크 없음 → **디코더 정확성 확정**.
- 검증 중 한 표본에서 특정 지점부터 PIL과 어긋나 처음엔 디코더 버그를 의심했다. 추적 결과
  Huffman 테이블 구성·AC/DC 로직·de-stuffing 모두 정상이었고, **그 지점이 실제 데이터 손상**이었다.

### 4. "디싱크 지점 = 실제 손상 위치"
- 디코더가 정확하므로, 정상 디코딩하다 어긋나는 지점이 곧 손상된 바이트다. 표본의 4개 Huffman
  테이블이 표준(Annex K)임도 확인했다. libjpeg는 이를 은폐할 뿐 정렬을 복원하지 못한다.

### 5. 바이트 오라클 + resync 설계
- **바이트 오라클**: 디싱크 지점 부근 바이트를 치환/삭제/삽입하며 "이후 clean run을 최대화"하는
  편집을 채택 → 단일바이트 손상은 정렬을 보존한 채 복구(밀림 없음).
- **resync-skip**: 단일바이트로 안 되면 재개 비트위치를 넓게 탐색해 손상 클러스터/구멍을 건너뜀.

### 6. 반복(중복) 버그 추적
- 일부 표본에서 **같은 내용이 세로로 주기적으로 중복** 출력됐다.
- 세그먼트별 비트 범위를 덤프하니, resync가 **스트림 맨 앞(비트 0)으로 재동기**해 앞부분을
  다시 디코딩하고 있었다(주기적 중복의 정체).
- 근본 원인: 디코더가 디싱크에서 멈출 때 `mcu_bit[frontier]`가 **미기록(0)**으로 남고, resync가
  `base_bit = mcu_bit[frontier] = 0`을 읽어 비트 0으로 점프. 두 변종 — (a) 세그먼트가 목표 범위
  전에 멈춤, (b) 비트레이트 체크가 `m += 1` *이후* return.
- 수정: 디코더가 디싱크 지점에서 **mcu_bit를 기록한 뒤** 멈추도록 → frontier 비트가 항상 유효.
  회귀 테스트(`test_recover_segments_strictly_increasing_bits`)가 세그먼트 비트의 단조 증가
  (= 재디코딩 없음)를 검증한다.

### 7. 정직성 결정 — 가짜 채움 거부
- 중복을 잡은 뒤에도, 재개점이 디싱크 지점과 거의 같은 비트(`db≈0`)인 resync는 어긋난 스트림을
  DC만 리셋해 그대로 다시 읽는 **가짜 복구**임을 발견 → `|db|≥24` 조건으로 거부.
- 남은 스트림 전체를 훑어 본 결과(scan_ahead), 일부 표본은 초기 손상 후 **비트의 95–99%를 소진**
  하고 재개 가능 지점이 0개 = **물리적 소실**임을 확인.
- 결정: 복구 불가 영역은 가짜로 채우지 않고 **회색으로 남긴다**(정직성). 이 때문에 회색 비율은
  공격적 채움보다 높지만, 보이는 내용은 진짜다.

### 8. 성능·철저함
- 손상 심한 파일에서 넓은 탐색을 반복해 한 파일에 수 분이 걸리는 경우 발견 → 파일당 `time_budget`
  안전장치 + 탐색 범위/시간 **옵션화**(`--fast`, `--time-budget`). 기본값은 복구율이 높은 철저 모드.
- 손상된 SOF(샘플링 0/컴포넌트ID 중복/0크기)는 거부(`SKIP_UNDECODABLE`)해 빈 이미지 크래시 방지.

## 기각된 가설 / 막다른 길

| 가설·시도 | 결과 |
|---|---|
| 회색 = 0-채움(데이터 손실) | ✗ — 0런 없음, 데이터 존재 |
| 가짜 마커 중화로 회색 제거하면 충분 | ✗ — 디싱크 garbage 노출. 회색=깨짐, 같은 원인 |
| 특정 지점부터 어긋남 = 디코더 버그 | ✗ — PIL 대조로 디코더 정확성 확정. 어긋남 = 실제 손상 |
| masking(db≈0) resync로 더 채우기 | ✗ — 가짜 반복 생성 → 거부 |
| `near`/`max_ops` 키우면 더 복구 | △ — 미미. 남은 회색 대부분은 물리적 소실(재개점 없음) |

## 사용한 방법·도구

- **프로토타입 스크립트**(gitignore된 `output/analysis/`, 참고용): 마커/엔트로피 스캔, 디코더 PIL
  대조, 세그먼트 비트범위 분석, 남은-스트림 복구가능성 스캔 등. 결론 검증에 쓴 일회성 도구다.
- **구현체**: `carver/jpegdecode.py`(디코더), `carver/resync.py`(복구 엔진).

## 결론

회색·깨짐은 **바이트 손상에 의한 비트스트림 디싱크의 두 얼굴**이다. 디싱크 지점을 정확히 짚어
정렬을 복원(바이트 오라클 / resync)하면 구조가 돌아온다. 물리적으로 소실된 영역은 정직하게 회색으로
남긴다. 색 캐스트·밝기 밴드·이미지 밀림은 별도 과제로 미뤘다. 결정은 [ADR 0001](../adr/0001-resync-recovery.md),
동작은 [spec 0002](../specs/0002-recover.md), 원리는 [reference](../reference/jpeg-entropy-coding.md) 참조.

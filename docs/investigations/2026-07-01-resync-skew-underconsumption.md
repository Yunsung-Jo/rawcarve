# 조사 기록 — 잔존 회색은 carve 정상·데이터 한계(skew는 재동기로 복구 불가)

- **날짜:** 2026-07-01
- **한 줄:** zero DC 리셋 복구 후에도 남는 회색을 진단 → carve 추출 정상(재추출 +0B)·output 신버전 확인, 잔존 회색은 데이터 소진 + 재동기가 특정 세그먼트에서 데이터를 폭식하는 디싱크(0x98265000 seg14 skew 0.61→1.69). `_resync_skip`이 clean run만 채점하고 skew를 무시하는 게 원인 → skew 채점 실험.
- **결론 문서:** [ADR 0004](../adr/0004-resync-dc-reset-recovery.md) 향후 고려사항(비트 과소비·데이터 소진을 데이터 한계로 명시). 선행 [resync-limit](2026-06-29-resync-limit.md)·[carve-eoi 3단계](2026-06-29-carve-eoi-discovery.md)의 교훈과 일치.

> **랩 노트(스냅샷).** [zero DC 리셋 조사](2026-07-01-resync-dc-reset.md) 후속. zero로 회색을 줄인 뒤 사용자 요청으로 "잔존 회색의 정확한 원인"을 진단한다. carve/데이터소진/재동기 세 축으로 갈랐다.

---

## 증상

사용자 제공 3샘플(zero 복구 후 undec 잔존): 0xB9ECA000 0.118, 0x98265000 0.319, 0xCE37C000 0.377. 요청: "carve 추출 문제인지, 데이터 소진인지, 재동기 실패인지, 놓친 부분이 있는지" 규명.

## 조사 과정 (가설 → 예측 → 증거)

### 1단계 — carve 추출 누락? → 기각

- **가설/이유:** 회색 = carve 가짜 EOI 조기종료(추출 누락, [ADR 0002](../adr/0002-carve-eoi-validation.md) 유형). 0xCE37C000이 용량 작아(1.4MB) 의심.
- **실험:** 기본 진단(파일크기·EOI위치·MCU당bit·끝0채움) + 현재 carve `jpeg_end`로 재추출한 크기를 output/jpeg와 대조. `scratchpad/diag_deep.py`.
- **증거:**
  - 3개 다 EOI@파일끝, 끝0채움 0B, MCU당 522~554bit(정상 범위). 0xCE37C000이 작은 건 **해상도가 작아서**(2592×1944, MCU 19764 vs 46464)지 누락 아님.
  - 현재 carve 재추출: 3개 다 **+0B**(구=신 동일).
  - disk_after(파일끝 이후 128KB stuffing) 0.85~0.94로 처음 carve 누락을 의심했으나, 이는 **윈도우를 크게 잡아 다음 파일 엔트로피를 섞어 본 착시**. jpeg_end는 EOI직후 4KB + `_next_header` 상한으로 정확 판별.
- **판단:** 기각. 3개 다 carve 정상 추출(진짜 EOI).

### 2단계 — output/jpeg 구버전? → 기각

- **가설/이유:** carve-eoi 커밋(ed5b278, 2026-06-29 04:56)이 output/jpeg 생성(2026-06-28 22:59)보다 6시간 늦어, output이 ADR 0002 적용 전 구버전일 가능성.
- **실험:** 보고서에서 "커졌다"고 기록된 246개 중 5개(0x96B61000 247KB→3MB 등)의 현재 output/jpeg 크기를 최신 carve 재추출 크기와 대조.
- **증거:** 5개 모두 output/jpeg가 이미 커진 크기(0x96B61000 3076080B = 재추출 3076080B, 0x9906A000·0x99C35000·0x9F549000·0x43AEE000 동일). 즉 신버전.
- **판단:** 기각. 커밋 시각은 코드 작성 후 지연 커밋일 뿐, output/jpeg는 이미 최신 carve 결과다(사용자 지적으로 확인). carve 재추출 불필요, zero 복구는 올바른 데이터 위에서 수행됨.

### 3단계 — skew 곡선: 2/3 재동기 폭식(디싱크)

- **가설/이유:** carve·구버전 기각 후 남은 원인은 데이터 소진 + 비트 과소비(skew). skew가 특정 지점부터 급증하면 디싱크(비트 과소비, 복구 여지), 처음부터 균일하면 데이터 특성(원본이 일부만 — carve-eoi 3단계 교훈).
- **실험:** `recover_trace`(recover 복제 + 세그먼트별 frontier·MCU%·BYTE%·skew 기록)로 3샘플 추적. `scratchpad/diag_skew.py`.
- **증거:**
  - 0xB9ECA000: skew 1.0~1.13 **균일**(급증 없음). BYTE 100%/MCU 88%. 디싱크 아님.
  - 0x98265000: seg0~13 skew **0.61**(데이터 절반만 쓰는 효율 구간), **seg14서 0.61→1.69 폭증**(seg_start 22177, frontier 22177→24497, BYTE 29.0→89.1%). 이후 1.47~1.69.
  - 0xCE37C000: seg2(0.48→1.28, seg_start 6059 BYTE 14.8→50.4%), seg5(→1.72, BYTE 64.7→93.4%) 급증.
- **판단:** 2/3(0x98265000·0xCE37C000)은 **특정 세그먼트에서 데이터를 폭식하는 디싱크**. `_resync_skip`이 clean run(재개 후 MCU 수)만 최대화하고 데이터 소비 효율(skew)을 무시해, "clean run은 길지만 데이터를 폭식하는" 나쁜 재동기 지점을 택한다. resync-limit은 누적 평균 rate로 못 잡았으나(정상 고비트레이트 오판), 세그먼트별 국소 skew 급증(BYTE 29→89%)은 명확한 신호다. → skew 채점 실험(4단계).

### 4단계 — skew 채점 재동기: 실패(모두 악화). skew는 재개 위치 기준이 못 됨

- **가설/이유:** 3단계에서 2/3가 특정 세그먼트에서 데이터 폭식. 재동기 후보를 clean run뿐 아니라 데이터 소비(bits = end_bit − resume_bit)로도 채점(run 동률이면 bits 최소)하면 폭식 지점을 피하고 데이터를 아낀다.
- **실험:** `skip_skew` monkeypatch — near 스캔 break 제거, best=(run 최대, 동률이면 bits 최소). cands=(캐리, zero) 유지. 5샘플 baseline(본체 zero) vs skew. `scratchpad/harness_skew.py`.
- **예측:** (실험 전, `scratchpad/exp_skew_prediction.md`) 디싱크 2개(0x98265000·0xCE37C000) undec 감소, 균일(0xB9ECA000)·가드(0xA44AE000·0x9906A000) 불변.
- **증거:** 전부 악화. 0x98265000 0.319→0.342(+0.023), 0xCE37C000 0.377→**0.629**(+0.252, resync 6→2), 0xB9ECA000 0.118→0.169, 가드 0xA44AE000 0.000→0.058·0x9906A000 0.000→0.059(**회귀**). 예측 완전 빗나감.
- **판단:** 기각. bits 최소 선호는 "우연히 짧게 맞은 나쁜 정렬"을 택해 재동기 품질을 떨어뜨린다(resync 수 감소, 가드 회귀). **seg14 폭식은 재개 위치의 문제가 아니라 그 구간 데이터의 특성**이다 — skew는 재동기로 고칠 신호가 아니었다.

## 기각된 가설 / 막다른 길

- **carve 누락 의심(disk_after stuffing 0.85~0.94) → 기각.** 파일끝 이후 stuffing을 128KB 윈도우로 재니 다음 파일 엔트로피가 섞여 높게 나왔다. 교훈: **파일끝 이후 stuffing은 윈도우를 크게 잡으면 다음 파일을 섞어 착시를 만든다** — jpeg_end의 4KB + `_next_header` 상한이 정확한 판별이고, 확실한 검증은 재추출 크기 대조다.
- **output 구버전 의심(커밋-생성 시각차 6h) → 기각.** 교훈: **커밋 시각 ≠ 파일 생성 시각.** 데이터 버전을 시각으로 추정하지 말고 실제 재생성물과 대조해 확인한다(커진 파일이 이미 커져 있으면 신버전).
- **skew(bits) 채점 재동기 → 실패(모두 악화, 가드 회귀).** 교훈: **재동기 후보는 clean run으로 채점해야 하며, 데이터 소비(bits/skew)는 품질 기준이 못 된다** — bits 최소는 나쁜 정렬을 선호한다. 비트 과소비(skew 급증)는 재개 위치로 고칠 디싱크가 아니라 그 구간의 데이터 특성이다. **누적 평균 rate(resync-limit)도, 국소 skew(이 실험)도 재동기 기준이 못 된다**는 3번째 확인.

## 사용한 방법·도구

- `scratchpad/diag_deep.py` — hole 심화(frontier·MCU%·BYTE%·skew·tail) + 디스크 대조(파일끝 이후 stuffing·다음 SOI).
- `scratchpad/diag_skew.py` — `recover_trace`(recover 복제)로 세그먼트별 skew 곡선 추적.
- 재추출 검증(jpeg_end로 크기 대조), output 구/신 판별(보고서상 커진 파일 대조).

## 결론

- 사용자 3샘플 회색 잔존의 원인은 **carve 정상 추출 + 데이터 소진/특성**이다. carve 누락(1·2단계 기각)도, 재동기 실패도 아니다. 셋 다 BYTE 100%·tail 1B로 추출 데이터를 완전히 소진했고 이미지의 62~88%만 채워졌다.
- **skew 급증(0x98265000 seg14 0.61→1.69)은 재동기로 고칠 디싱크가 아니라 데이터 특성**이다. skew 채점(4단계)이 오히려 악화시켜 이를 확인했다 — [resync-limit](2026-06-29-resync-limit.md)의 rate 강화 실패, [carve-eoi 3단계](2026-06-29-carve-eoi-discovery.md)의 "skew는 데이터 특성" 교훈과 일치한다.
- 결론: 이 회색은 **데이터 한계로 복구 불가**하며, zero DC 리셋([resync-dc-reset 조사](2026-07-01-resync-dc-reset.md))이 이 파일들에 대한 최선이다. 비트 과소비는 별도 레버로 추적하지 않는다(3번 확인된 막다른 길).
- 부수 성과: **carve/데이터 버전 진단 방법**을 확립했다 — 재추출 크기 대조(carve 누락 판별), 커진 파일 대조(데이터 신/구버전), 세그먼트별 skew 곡선(디싱크 vs 데이터 특성).

# JPEG 복구 파이프라인 개발 일지

**날짜:** 2026-05-25  
**작성자:** 조윤성 (with Claude Sonnet 4.6)

---

## 1. 기획 배경

전날 `carve.py`로 USB 이미지에서 JPEG를 추출했는데, 이미지 뷰어로 열면 회색으로 보이는 파일이 다수 있었다. HxD로 열어보니 데이터가 일부 살아있는 것 같았고, 완전히 버리기엔 아까운 파일들이었다.

> "추출된 jpeg를 이미지 뷰어로 열어보면 회색으로 보이는 문제가 있어. HxD로 이미지를 열어보면 데이터가 일부 손상되었지만 살아있는 것 같아. @output\jpeg\ 에 추출된 이미지를 분석해서 이미지가 회색으로 보이는 원인을 분석해줘."

분석 결과 회색 원인은 단일하지 않았다. BAD_STUFF(스캔 데이터 내 마커 위반), MARKER_BYTE_FLIP(헤더 마커 비트 반전), GRAY_MCU(회색 MCU 패턴 주입), ZERO_FILL(스캔 데이터 영값 채움) 등 복수의 손상 패턴이 확인됐다. 이를 원인별로 진단하고 각각에 맞는 복구 전략을 적용하는 파이프라인을 만들기로 했다.

---

## 2. 주요 결정

### BAD_STUFF 복구 전략: 절단 vs FF→00 교정

초기 설계는 BAD_STUFF 첫 발생 위치에서 파일을 잘라내는 방식(truncation)이었다. RST(리스타트) 마커가 있으면 그 직전까지만 살릴 수 있기 때문이다.

그런데 실제 추출 파일 200여 개를 분석하니 **RST 마커가 0개**였다. 일반 카메라 JPEG는 RST 마커를 사용하지 않는 경우가 많다. RST 없이 스캔 데이터가 손상되면 절단 이후 데이터는 전부 회색으로 채워지므로 절단 방식의 의미가 없었다.

대안으로 나온 것이 **FF→00 바이트 교정**이다. 디스크 손상 패턴을 분석하니 `FF EC`, `FF CB` 같은 위반 바이트들이 원래 `FF 00`이었을 가능성이 높았다. 물리 디스크에서 비트 오류는 바이트를 삭제하는 게 아니라 값을 바꾸는 방식으로 나타나기 때문이다. `FF 00 EC`였다가 `FF EC`로 줄어든 게 아니라, `FF 00`이 `FF EC`로 변한 것으로 봐야 한다는 결론이었다.

이 방식으로 교정하면 JPEG 디코더의 싱크가 다시 맞춰지고, 이후 손상된 블록은 `libjpeg`가 회색(128, 128, 128)으로 채운다. 이 회색 블록들을 별도로 감지해서 이웃 블록으로 보간하면 눈에 보이는 복구가 가능하다.

최종 전략:
1. **BAD_STUFF**: FF→00 교정 우선 → 디코딩 후 손상 블록 보간
2. 교정 후에도 90% 이상 손상이면 원본 데이터로 강제 디코딩 폴백
3. **GRAY_MCU / 기타**: 강제 디코딩 + 보간만 적용
4. **MARKER_BYTE_FLIP**: 헤더 마커 교정 후 동일 파이프라인

### 스토리지 파편화 가설 검토

> "스토리지 파편화가 원인일 가능성도 있을까? 4kb, 16kb 단위로 데이터가 오염되어 있을 수 있잖아. 분석해줘."

실제 파일들의 BAD_STUFF 발생 간격을 분석한 결과, 오염 구간 길이가 4 KB/16 KB 클러스터 경계와 유의미하게 정렬되지 않았다. 82.6%의 BAD_STUFF 간격이 10 바이트 미만이었고, 파편화에 의한 클러스터 단위 오염 패턴이 아니었다. 스토리지 파편화는 주요 원인에서 제외했다.

### 손상 블록 임계값

복구 가능 여부를 결정하는 임계값을 90%로 설정했다. 90% 이상 손상되면 `SKIP_TOO_DAMAGED`를 반환하고 파일을 생성하지 않는다. 10%~90% 구간은 보간을 적용하고, 10% 미만 손상은 보간 없이 그대로 저장한다(보간 비용 대비 효과가 미미하기 때문).

### subagent-driven development 채택

설계가 확정된 후 구현은 subagent-driven development 방식으로 진행했다. 6개 태스크를 순서대로 신선한 서브에이전트가 구현하고, 각 태스크마다 스펙 컴플라이언스 리뷰 → 코드 품질 리뷰 2단계를 거쳤다.

## 3. 사용자 피드백 / 주요 프롬프트

> "RST 마커는 여러 개 있는 거 아니야? RST 마커가 없는 부분 뒤로는 전부 버리는 거야?"

이 질문이 절단 방식의 한계를 정확히 짚었다. RST 마커가 실제로 없다는 것을 확인한 뒤 설계를 바꿨다.

> "RST 마커가 없어도 손상된 이미지를 살릴 수 있는 방법은?"

FF→00 교정 + 강제 디코딩 + 보간이라는 최종 전략의 출발점이 된 질문이다.

> "추가적인 분석 하나만 더 해줘. 너가 제안한 방법은 FF EC를 FF 00으로 변경하는 거잖아? FF 00 EC였을 가능성은 없을까?"

삭제 모델(바이트가 사라진 경우)과 교체 모델(바이트 값이 바뀐 경우) 중 어느 쪽인지를 확인하는 질문이었다. 실제 오염 패턴 분석으로 교체 모델이 맞다는 결론을 내렸다.

## 4. 구현 중 발견된 문제와 수정

### 단색 JPEG에는 FF 00 스터핑 바이트가 없다

테스트 작성 시 단색(solid color) JPEG를 사용했는데, 단색 이미지의 DCT 계수는 균일해서 `FF 00` 시퀀스가 스캔 데이터에 거의 생성되지 않는다. BAD_STUFF 테스트에서 손상시킬 `FF 00`을 찾지 못해 오류가 발생했다. 노이즈/그레디언트 이미지로 교체해서 해결했다.

### CANDIDATE_FIXES 마커가 SOF로 오파싱되는 버그

`_parse_header`에서 SOF 세그먼트를 파싱할 때 `0xCB`(SOF11), `0xC3`(SOF3), `0xC5`(SOF5)도 `_SOF_MARKERS` 집합에 포함되어 있었다. 그런데 이 마커들은 동시에 `CANDIDATE_FIXES`의 키이기도 하다. DQT(`0xDB`)가 SOF11(`0xCB`)로 반전된 파일을 파싱하면 DQT 페이로드를 SOF 크기/채널 필드로 읽어버려서 진단 결과가 틀렸다.

`mb not in CANDIDATE_FIXES` 조건을 추가해서 수정했다.

### MARKER_BYTE_FLIP + FALSE_POSITIVE 이중 cause 상태

`diagnose()`에서 MARKER_BYTE_FLIP을 감지한 뒤 `scan_start == -1`이면 FALSE_POSITIVE도 `append`했다. 결과적으로 `['MARKER_BYTE_FLIP', 'FALSE_POSITIVE']`라는 모순 상태가 생겼다. `recover_file`에서 MARKER_BYTE_FLIP을 우선 처리하므로 FALSE_POSITIVE 건너뜀이 동작하지 않았다.

`r.causes = ['FALSE_POSITIVE']`로 교체(append 대신 재할당)해서 수정했다.

### 코드 품질 리뷰에서 발견된 문제들

- `bytes(patched)` 이중 변환: `_recover_marker_flip`에서 같은 bytearray를 두 번 `bytes()`로 변환하고 있었다. `fixed` 변수로 한 번만 변환하도록 수정.
- MARKER_BYTE_FLIP 실패 시 BAD_STUFF 폴백 없음: 마커 교정 후 여전히 너무 손상된 경우, 같은 파일에 BAD_STUFF도 있다면 BAD_STUFF 복구를 시도할 기회가 없었다. `recover_file`에 폴백 로직 추가.
- `_damage_pct`가 디코딩 완전 실패 시 `0.0` 반환: 정상 파일(손상 0%)과 디코딩 불가 파일이 같은 값으로 표시되어 CSV가 오해를 줄 수 있었다. `1.0`으로 변경.
- 스모크 테스트에서 `recover.py` 상대 경로 사용: `cwd`에 따라 테스트가 깨질 수 있어서 절대 경로로 수정.

## 5. 최종 결과

- `carver/diagnosis.py`: 6개 손상 범주 진단 (FALSE_POSITIVE / MARKER_BYTE_FLIP / BAD_STUFF / GRAY_MCU / ZERO_FILL / CLEAN)
- `carver/recovery.py`: 손상 블록 감지(8×8 IDW 보간), 원인별 복구 함수, `recover_file()` 오케스트레이터
- `recover.py`: CLI 진입점, `report.csv` 7컬럼 리포트 생성
- 테스트: 54개 전체 통과 (진단 7개 + 복구 14개 + 기존 33개)

```bash
# 사용 예시
python carve.py usb.img -o output/
python recover.py output/jpeg/ -o output/jpeg_recovered/
# output/jpeg_recovered/report.csv 에 진단·복구 결과 기록
```

## 6. 알려진 한계

- **RST 마커 없는 JPEG의 후반부**: BAD_STUFF 이후 구간은 보간으로 채우지만, 원본 픽셀 정보가 완전히 소실된 블록은 이웃 색상으로 근사할 뿐이다. 심하게 손상된 파일은 `SKIP_TOO_DAMAGED`(90% 임계값)로 건너뛴다.
- **GRAY_MCU 패턴 의존성**: `01 45 00 14 50` 5바이트 시퀀스 반복으로 감지하는데, 다른 인코더가 생성한 회색 MCU 패턴은 감지 못할 수 있다.
- **AVI 복구 없음**: 현재는 JPEG만 복구한다. AVI 내 손상된 JPEG 프레임도 개별 복구 대상이 될 수 있으나 구현하지 않았다.
- **다중 마커 반전 미처리**: `CANDIDATE_FIXES`에 없는 마커 반전, 또는 동일 파일에서 두 곳 이상 반전된 경우는 복구하지 못한다.
- **보간 품질**: 현재 IDW(역거리 가중) 방식은 이웃 블록의 평균 색상만 사용한다. 텍스처나 엣지 정보는 반영하지 않아 경계 부근 복구 품질이 낮다.

# rawcarve 개발 일지

**날짜:** 2026-05-24  
**작성자:** 조윤성 (with Claude Sonnet 4.6)

---

## 1. 기획 배경

어린 시절 사진과 영상이 저장된 손상된 USB가 있었다. ddrescue로 이미지 파일은 떴지만(`usb.img`, 3.35 GB), 그 안에 담긴 JPEG와 AVI 파일을 꺼내는 것이 목표였다.

기존 도구를 써봤지만 손상된 부분이 많아서인지 제대로 복구되지 않았다. 그래서 직접 구현하기로 했다. 동작 방식을 정확히 이해하고 싶기도 했고, Claude Code를 실전에서 써보고 싶은 이유도 있었다.

디스크 이미지가 손상되어 있기 때문에 파싱 중 쓰레기 값이 나올 수 있고, 오류가 생겨도 프로그램이 도중에 중단되어서는 안 된다는 제약이 있었다.

---

## 2. 브레인스토밍 — 주요 결정

Claude와 설계를 논의하며 아래 사항들을 하나씩 결정했다.

**최초 요청:**
> "ddrescue 등으로 복구한 손상된 디스크 이미지(.img)에서 이미지 및 영상 파일을 추출하는 프로그램을 만들고 싶어. 이미지는 jpeg, 영상 파일은 avi 파일만 이미지에 포함되어 있어. 디스크 이미지가 손상되어 있기 때문에 쓰레기 값이 있을 수 있어. 초기 프로젝트 세팅도 하고 싶어. CLAUDE.md 나 커밋 메시지 규칙 등을 정하자. @usb.img 를 참고해서 구현해도 좋아. 다만 @usb.img 파일과 output 파일은 커밋되지 않게 제외하는 게 좋을 것 같아."

### 언어

> **Python** 선택. 빠른 프로토타이핑, `mmap` 기본 지원, tqdm 생태계 활용.

### 스캔 방식

`mmap`으로 디스크 이미지를 메모리 매핑한 뒤 `bytes.find()`로 시그니처를 탐색한다. 파일 전체를 한 번에 메모리에 올리지 않고 OS 페이지 캐시를 통해 필요한 부분만 로드되므로 3 GB 이상도 처리 가능하다.

### AVI 파일 경계 결정

> **RIFF 청크 크기 + 최대 크기 제한 폴백** 선택.

RIFF 헤더의 4~7바이트(little-endian uint32)가 파일 크기를 담고 있다. 유효하면 그대로 사용하고, 이상값이면 다음 시그니처 위치를 경계로 쓴다. `--max-avi-size` 옵션으로 상한도 걸 수 있다.

`RIFF` 시그니처는 WAV, AVI 등 여러 포맷이 공유하므로, offset+8~11이 `AVI `인지 추가로 검증해 WAV를 걸러낸다.

### 출력 파일명

> **타입별 서브폴더 + 16진수 오프셋 이름** 선택.

```
output/jpeg/0x01A3F000.jpg
output/avi/0x03B20000.avi
```

파일명에 디스크 이미지 내 원본 위치가 담겨 있어 나중에 추적하기 쉽다.

### 진행률 표시

> **tqdm 진행률 바 + 파일별 발견 로그** 선택.

tqdm 바로 전체 진행률을 표시하고, `tqdm.write()`로 각 파일 추출 결과를 줄별로 출력한다. 두 출력이 서로 겹치지 않는다.

### CLI

argparse로 구성. 필수 인자는 이미지 파일 경로, 선택 옵션은 `-o/--output`, `--max-avi-size`, `--save-thumbnails`.

### 커밋 메시지

> "커밋 메시지는 한글로 작성하면 하고 Co-Authored-By 작성 및 body도 작성했으면 좋겠어."

Conventional Commits 형식, **한글** 작성, body와 `Co-Authored-By: Claude Sonnet 4.6` 포함.

## 3. 썸네일 문제 — 설계 재논의

디지털 카메라 JPEG에는 미리보기용 **내장 썸네일**이 APP1/EXIF 세그먼트 안에 포함된다. 스캐너가 시그니처(`FF D8 FF`)를 탐색하면 부모 JPEG의 시작과 내장 썸네일의 시작을 모두 히트로 잡는다.

이를 어떻게 처리할지 논의가 있었다.

**썸네일 문제 제기:**
> "궁금한 게 있어. 디지털 카메라로 촬영한 거라 그런지 이미지 크기가 작은 것도 있고 큰 것도 있었어. 아마 미리보기용 썸네일에 해당하는 것 같기도 한데, 이건 어떻게 처리할거야?"

### 시도 1: --min-jpeg-size 크기 기반 필터링

처음에는 특정 크기 미만의 JPEG를 썸네일로 간주하는 `--min-jpeg-size` 옵션을 설계에 넣었다.

> "--min-jpeg-size 미만이면 썸네일로 간주는 그닥 좋은 방법이 아닌 것 같아. 보니까 1,024 bytes 보다는 훨씬 큰 것 같고.."
>
> "범위 기반만 유지, --min-jpeg-size 제거하자"

옳은 지적이었다. 썸네일 크기는 카메라 기종마다, 사진 크기마다 다르기 때문에 크기 임계값은 정할 수 없다. `--min-jpeg-size`는 설계에서 제거했다.

### 채택: 범위 기반 감지

부모 JPEG의 정확한 범위(시작 오프셋 ~ 끝 오프셋)를 추출 직후 누적 관리한다. 새로운 JPEG 히트의 오프셋이 이미 추출된 파일 범위 안에 들어오면 내장 썸네일로 분류한다.

이 방식이 동작하려면 **부모 JPEG의 끝을 정확히 계산하는 것**이 전제 조건이다.

## 4. JPEG 끝 탐지 — 핵심 기술적 도전

### 문제

JPEG 파일은 `FF D8`(SOI)로 시작하고 `FF D9`(EOI)로 끝난다. 가장 단순한 구현은 `FF D9`를 찾을 때까지 탐색하는 것이다.

그런데 APP1/EXIF 내 내장 썸네일 **자체도** 완전한 JPEG이므로 `FF D9`로 끝난다. 단순 탐색은 이 썸네일의 `FF D9`에서 멈춰 부모 파일을 잘라버린다.

> "부모 파일의 정확한 계산이 가능할까? 예를 들면 JPEG 파일은 FF D8로 시작해서 FF D9로 끝나잖아? 그러면 부모의 FF D8을 찾고 썸네일의 FF D9를 만나서 종료되는 케이스가 생길 수 있을텐데 그렇지 않게 구현할 수 있지? 파일의 진짜 끝부분을 찾아낼 수 있어야 해"

### 해결: JPEG 세그먼트 파싱

JPEG는 `FF XX` 마커와 2바이트 길이 필드로 구성된 세그먼트 구조를 갖는다. 길이 필드를 읽으면 세그먼트 전체를 건너뛸 수 있다.

```
FF E1 (APP1) | 길이(2B big-endian) | [EXIF 데이터 + 내장 썸네일]
```

APP1 세그먼트의 길이 필드를 읽고 그만큼 건너뛰면, 썸네일 안의 `FF D9`를 전혀 보지 않고 다음 외부 마커로 이동한다. 이후 SOS(`FF DA`) 이후의 스캔 데이터에서 진짜 `FF D9`를 찾는다.

**SOS 이후 스캔 데이터 처리 규칙:**
- `FF 00` — 스터프드 바이트(이스케이프), 건너뜀
- `FF D0`~`FF D7` — RST(재시작) 마커, 건너뜀  
- `FF FF` — 필 바이트(JPEG 스펙 §B.1.1.2), 건너뜀
- `FF D9` — 진짜 EOI, 파일 끝

## 6. 리뷰에서 발견된 버그 및 수정

### Bug 1: SOS 스캔 데이터 경로 미테스트

코드 품질 리뷰에서 JPEG SOS 이후 스캔 데이터 처리 경로에 대한 테스트가 없다는 지적이 나왔다.

추가된 테스트:
- `test_jpeg_end_with_scan_data` — `FF 00` 스터프드 바이트 처리 검증
- `test_jpeg_end_scan_data_with_rst` — `FF D0`~`FF D7` RST 마커 처리 검증

### Bug 2: FF FF 필 바이트 처리 누락

JPEG 스펙 §B.1.1.2에 따르면 마커 앞에 `FF FF` 필 바이트가 올 수 있다. 세그먼트 헤더 루프에서 이를 처리하지 않아 이상한 마커 번호로 읽히는 버그가 있었다.

```python
# 수정 전: FF FF를 알 수 없는 마커로 처리
# 수정 후: FF FF를 만나면 건너뜀
if mb == 0xFF:
    pos += 1
    continue
```

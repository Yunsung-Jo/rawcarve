# recover.py

- **날짜:** 2026-06-03
- **상태:** Accepted

---

## 개요

`carve.py`가 추출한 손상 JPEG를 **resync 엔진**으로 복구하는 도구.
바이트 단위로 디싱크(비트 정렬 어긋남) 지점을 짚어 정렬을 복원하고,
복구 가능한 영역만 복구한 뒤 전체 결과를 `report.csv`로 저장한다.
설계 근거는 [ADR 0001](../adr/0001-resync-recovery.md) 참조.

## 인터페이스

| 이름 | 설명 | 타입 | 기본값 |
|------|------|------|--------|
| `input` | 입력 디렉토리 (`output/jpeg/`) | positional | — |
| `-o, --output` | 출력 디렉토리 | str | `<input>_recovered` |
| `-q, --quality` | 복구본 JPEG 품질 | int | `95` |
| `-j, --jobs` | 병렬 프로세스 수 (`0`=CPU 수, `1`=순차) | int | `0` |
| `--fast` | 빠른 모드(부분 복구 감수); 미지정 시 철저 | flag | 철저 |
| `--time-budget` | 파일당 시간 상한(초, `0`=무제한) | float | 철저 90 / fast 20 |

**철저↔속도 조절**: `_resync_skip`의 탐색 범위로 복구율과 속도가 갈린다. 철저(기본)는
`resync_near=300000` + 전체 스트림 거친 스캔(`full=True`)으로 먼 구멍까지 건너뛴다(복구율↑, 느림).
`--fast`는 `near=160000` + 전체 스캔 끔(빠름). `time_budget`은 심손상 파일의 비용 폭발을 막는
안전장치다(초과 영역은 회색). 단 추출 파일이 큰 경우(수 MB) 기본 budget으로는 시간이 부족해
복구가 도중에 멈춰 회색이 남을 수 있으므로 `--time-budget 0`(무제한)을 권장한다.

## 출력

```
<output>/
  report.csv              # 파일별 복구 결과 (4종 모두 기록)
  recovered/              # 재인코딩된 복구본 (RECOVERED)
  clean/                  # 손상 없던 원본 복사 (CLEAN)
  skip_undecodable/       # 디코드 실패 원본 복사 (SKIP_UNDECODABLE)
  error/                  # 워커 예외 발생 원본 복사 (ERROR)
```

`main`이 실행 전 네 서브폴더를 미리 생성하므로,
해당 분류 파일이 없더라도 폴더는 항상 존재한다.
`report.csv`는 최상위에 위치하며 4종 분류를 모두 기록한다.

`report.csv` 컬럼:

| 컬럼 | 설명 |
|------|------|
| `filename` | 원본 파일명 |
| `action` | `RECOVERED` / `CLEAN` / `SKIP_UNDECODABLE` / `ERROR` |
| `gray_before` / `gray_after` | 복구 전/후 회색(평탄·무채색) 픽셀 비율 (0.0–1.0) |
| `ops` | 적용한 편집·재동기 총 횟수 |
| `sub` / `del` / `ins` | 바이트 치환 / 삭제 / 삽입 횟수 |
| `resync` | 비트위치 재동기(세그먼트) 횟수 |
| `hole` | 복구 불가로 중단한 횟수 |
| `image_size` | SOF 해상도 (`WxH`) |

## 파이프라인

1. **JPEG 목록 수집** (`recover.py`) — 입력 디렉토리에서 `*.jpg`를 정렬한다.
2. **복구** (`carver/resync.py::recover_file`) — 파일마다:
   - `carver/jpegdecode.py`로 디코더 구성(3-component baseline만 지원, 아니면 `SKIP_UNDECODABLE`).
   - 전체 디코드로 `gray_before` 측정.
   - `recover()`가 디싱크 지점마다 바이트 오라클(치환/삭제/삽입) → 실패 시 resync-skip을 적용.
   - 복구본을 RGB로 렌더 → JPEG로 재인코딩 저장. `gray_after`·편집 통계 반환.
3. **report.csv 작성** (`recover.py`).

## 복구 원리 (알고리즘)

> 배경 지식(엔트로피 코딩·디싱크가 회색/깨짐을 만드는 이유)은
> [JPEG 엔트로피 코딩과 디싱크 원리](../reference/jpeg-entropy-coding.md) 참조.

### 출발점: 디싱크 지점 = 손상 위치

자작 디코더는 표준 클린 JPEG과 일치함이 검증됐다. 따라서 **이 디코더가 정상 디코딩하다가
어긋나는(디싱크) 지점이 곧 실제 바이트 손상 위치**다. libjpeg는 손상을 은폐할 뿐
정렬을 복원하지 못하지만, 우리는 손상 지점을 정확히 짚어 정렬 자체를 되돌린다.

### 디싱크 탐지: plausibility

`carver/jpegdecode.py`의 디코더는 다음 중 하나라도 발생하면 그 MCU에서 멈추고
그 지점(frontier)의 비트위치·DC예측값을 정확히 보고한다:

| 신호 | 의미 |
|------|------|
| 무효 Huffman 코드 | 비트 정렬이 깨져 테이블에 없는 코드를 만남 |
| 계수 dequant 오버플로 (DC>1400, AC>6000) | 디싱크로 비현실적으로 큰 계수가 나옴 |
| **비트레이트 초과** (MCU당 평균의 4배) | 코드 길이는 유효하나 비트를 폭식 = "그럴듯하지만 어긋난" 디싱크 |

비트레이트 신호가 핵심이다. 잘못된 정렬에서도 우연히 유효한 코드가 한참 이어질 수 있는데,
그런 경우 MCU당 소비 비트가 비정상적으로 커진다. 이를 잡아 가짜 정렬을 걸러낸다.

### 복구 두 갈래

각 디싱크 지점 `m_d`에서:

**1) 바이트 오라클 (`_best_edit`)** — 단일바이트 손상 복구, 정렬 보존(밀림 없음).

`m_d` 부근 바이트를 한 개씩 **치환(256가지)·삭제·삽입**해보고, 각 후보에 대해
"그 직전 안전 지점부터 몇 MCU나 깨끗하게 디코딩되는가(clean run)"를 측정한다.
clean run을 최대화하는 편집을 채택한다. run이 상한(maxW)에 도달하면 즉시 채택(early-exit).
→ 디스크 비트플립 등 단일바이트 손상이 여기서 원래대로 복원되며, 비트 정렬이 그대로
유지되므로 **이미지 밀림이 생기지 않는다.**

**2) resync-skip (`_resync_skip`)** — 다중바이트 손상·구멍 건너뛰기.

단일바이트로 안 되면(손상 클러스터/섹터 손상), 재개 비트위치를 **넓게 탐색**한다:
가까운 범위(byte 정렬, 8비트 간격)부터 시작해, 못 찾으면 남은 스트림 전체를 거칠게(64비트) 훑어
clean run이 충분히 긴 재개점을 찾는다. 찾으면 그 지점부터를 새 **세그먼트**로 등록하고,
DC 예측값은 직전 값을 이어받는다(밝기 밴드는 보류 항목).

- **masking 거부:** 재개점이 `m_d`와 거의 같은 비트(`|db| < 24`)면, 이는 "어긋난 스트림을
  DC만 리셋해 그대로 다시 읽는" **가짜 복구**다. 거부한다.
- **구멍은 회색 유지:** 어떤 재개점도 충분히 길게 디코딩되지 않으면(물리적 소실), 그 영역은
  채우지 않고 회색으로 남긴다.

byte 편집은 우선, 안 되면 resync. 둘 다 안 되면 그 지점에서 종료(이후는 회색).

### 세그먼트 모델과 좌→우 처리

복구 상태는 **세그먼트 리스트** `(시작MCU, 시작비트, DC예측)`로 표현된다. 첫 세그먼트는
`(0, 0, [0,0,0])`. resync마다 세그먼트가 하나 추가된다. 렌더 시 각 세그먼트를 순서대로
디코딩해 전체 이미지를 만든다.

**항상 좌→우(작은 MCU→큰 MCU)로 처리**하므로, 바이트 편집·세그먼트 추가는 언제나 현재
frontier *이후*에만 일어난다. 따라서 이미 확정된 앞쪽 세그먼트의 비트위치는 영향받지 않는다.

> **회귀 주의 (반복/중복 버그):** 과거 버그는 디싱크 후 `mcu_bit[frontier]`가 미기록(0)으로
> 남아, resync가 `base_bit=0`을 읽고 **스트림 맨 처음으로 재동기 → 앞부분을 반복 디코딩**해
> 같은 내용이 주기적으로 중복됐다. 디코더가 디싱크 지점에서 **mcu_bit를 기록한 뒤 멈추도록**
> 고쳐 frontier 비트가 항상 유효하게 했다. `tests/test_resync.py`의
> `test_recover_segments_strictly_increasing_bits`가 세그먼트 비트의 단조 증가(=재디코딩 없음)를
> 검증해 이 회귀를 막는다.

### 무엇을 복구하고 무엇을 남기나

- **복구:** 비트 정렬(→ 구조·디테일). 회색·깨짐을 실제 내용으로 되돌린다.
- **보류(미해결):** 색 캐스트, resync 지점의 밝기 밴드(DC 리셋), 이미지 밀림. 구조 복원에 집중한다.
- **남김:** 물리적으로 소실된 영역은 가짜로 채우지 않고 회색으로 둔다(정직성).

## 미해결(별도 과제)

색 캐스트, resync 지점의 밝기 밴드(DC 리셋), 이미지 밀림은 복구 대상이 아니다.
resync 엔진은 **구조 복원과 회색 제거**에 집중한다.

후속 조사: 데이터가 남았는데 `_resync_skip`이 DC 예측값을 직전 값으로 잇기만(캐리) 해서
재동기에 실패하는 경우(DC=0으로 리셋하면 더 복구되나 색이 틀어짐)와, 어긋난 진행이 데이터를
과소비하는 **비트 폭식**(MCU당 누적 평균 비트레이트 상한으로는 탐지 안 됨)은
[investigation 2026-06-29](../investigations/2026-06-29-resync-limit.md) 참조.

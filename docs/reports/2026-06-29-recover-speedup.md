# 보고서 — recover 속도 8.2배: 1시간 20분 → 9분 43초 (복구 결과 불변)

- **날짜:** 2026-06-29
- **대상/범위:** 손상 JPEG 복구 도구 `recover.py`의 변경 전(커밋 `8bbc106` 시점) vs 변경 후(`63ad396`). 동일 입력 822개, 동일 조건(시간 제한 없음, 12개 병렬).
- **한 줄 요약:** 복구 처리를 8.2배 빠르게 했고(전체 1시간 20분 → 9분 43초), 복구 결과물(사진)은 822개 모두 **한 픽셀도 바뀌지 않았다**.
- **관련 문서:** [ADR 0003 성능 최적화 결정](../adr/0003-recover-perf-optimization.md), [조사 기록(프로파일링)](../investigations/2026-06-29-recover-perf-profiling.md), [spec 0002 recover](../specs/0002-recover.md)

> 결론 지향 문서. 측정된 결과·수치만 담는다. 분석 과정·막다른 길은 조사 기록에 있다.

## 1. 한눈에 보기

> **용어:** *복구* = 손상된 사진 파일(JPEG)을 다시 볼 수 있게 고치는 처리. *무손실(여기서)* = 처리 *속도*만 개선하고 복구된 사진 자체는 똑같이 유지한다는 뜻.

손상 사진 822장을 복구하는 전체 작업이 **약 1시간 20분 → 9분 43초**로 단축됐다(**8.2배**). 동시에 복구된 사진은 822장 전부 변경 전과 **완전히 동일**하다(아래 "무손실").

| 지표 | 변경 전 | 변경 후 | 효과 |
|------|--------:|--------:|------|
| 전체 처리 시간(822장, 12병렬) | 약 80분 | **9분 43초** | **8.2배 빠름** |
| 복구 결과물 동일성 | — | 822/822 동일 | **품질 영향 0** |
| 가장 무거운 사진 1장(단일 처리) | 187.7초 | 28.5초 | 6.6배 빠름 |

## 2. 배경·목적

`recover.py`는 손상된 사진을 한 장씩 분석해 고친다. 전체 822장 처리에 1시간이 넘게 걸렸고, 처리 중 컴퓨터의 다른 작업도 느려지는 현상이 있었다.

목적은 **복구 품질을 전혀 건드리지 않고 처리 속도만 높이는 것**이었다. 즉 변경 후 복구된 사진이 변경 전과 픽셀 단위로 같아야 한다는 제약을 두었다.

## 3. 방법

- **속도:** 변경 전/후를 동일 입력(822장)·동일 조건(시간 제한 없음, 12개 병렬)으로 실행해 전체 소요 시간을 비교했다. 변경 전 시간은 그 작업이 남긴 결과 폴더의 파일 생성 시각(첫 파일 ~ 최종 기록 파일)으로 확인했다.
- **무손실(품질 불변):** 변경 전후의 복구 결과를 비교했다. ① 대표 6장은 사진 데이터의 지문값(sha256 — 내용이 1비트라도 다르면 값이 달라지는 체크값)이 일치하는지, ② 822장 전체는 복구 기록표(`report.csv`)의 모든 항목(복구 여부·복원 못 한 영역 비율·편집 횟수 등)이 일치하는지 확인했다.

## 4. 결과

### 4-1. 속도: 전체 8.2배

| | 변경 전 | 변경 후 |
|---|--:|--:|
| 822장 전체(12병렬) | 4,771초(79.5분) | 583초(9.7분) |

→ **8.2배.** 1시간 20분 걸리던 작업이 10분 안에 끝난다.

처리가 무거운 사진일수록 효과가 컸다(한 장씩 단독 측정):

| 사진 | 변경 전 | 변경 후 | 배수 |
|------|--------:|--------:|-----:|
| 가벼운 편 | 1.24초 | 0.91초 | 1.4배 |
| 중간 | 12.18초 | 3.24초 | 3.8배 |
| 무거운 편 | 68.04초 | 10.94초 | 6.2배 |
| 가장 무거움 | 187.72초 | 28.50초 | 6.6배 |

→ 의미: 시간을 많이 잡아먹던 무거운 사진이 가장 크게 빨라져, 전체 작업 시간이 그만큼 줄었다.

### 4-2. 무손실: 복구 결과물 822장 전부 동일

- 대표 6장: 사진 데이터 지문값(sha256) **전부 일치**.
- 822장 전체: 복구 기록표의 모든 항목 **전부 일치**(복구 분류 — 복구됨 557 / 원본 양호 143 / 복구 불가 122 — 도 변경 전과 동일).

→ 의미: 속도만 빨라졌고 **복구된 사진은 바뀌지 않았다**. 정상 결과를 망칠 위험이 없다.

### 4-3. 부수 효과: 컴퓨터 부하 진동 해소

변경 전에는 12개 작업 단위(병렬 프로세스)의 CPU 사용률이 **8% ↔ 0%로 계속 진동**했다. 원인은 *메모리 대역폭*(메모리를 읽고 쓰는 통로의 속도 — 저장 용량과는 다른 자원) 포화였다. 작업들이 같은 데이터를 불필요하게 대량 복사하면서 메모리 통로를 막아, CPU가 일을 못 하고 대기하던 것이다. 이 복사를 없애자 진동이 사라졌고(변경 후 작업 관리자에서 확인), 다른 앱이 느려지던 현상도 함께 해소됐다.

→ 이것이 전체 배수(8.2배)가 사진 1장 단독 배수(6.6배)보다 **더 큰** 이유다: 작업들이 서로의 메모리 통로를 뺏지 않게 되어 12개가 깨끗하게 병렬로 돌게 됐다.

## 5. 결론·권고

- 복구 처리를 **8.2배** 빠르게 했고(1시간 20분 → 9분 43초) 복구 결과물은 **완전히 동일**하다. 품질 손실 없이 순수하게 빨라진 개선이다.
- 무엇을 바꿨나(요약): 복구 도중 한 글자(바이트)만 바꿔 시험할 때마다 사진 데이터 전체(약 3 MB)를 통째로 복사하던 낭비를 없앴고, 비트 단위로 한 칸씩 읽던 부분을 한 번에 읽도록 바꿨다. 둘 다 결과는 동일하고 속도만 개선된다.
- 권고: 변경을 그대로 채택. 추가 가속(품질을 약간 희생하는 방식)은 본 작업의 "품질 불변" 원칙과 맞지 않아 적용하지 않았다.

## 6. 한계

- **변경 전 80분은 추정 하한이다.** 직접 재측정한 값이 아니라, 변경 전 작업이 남긴 결과 폴더의 첫 파일~최종 파일 생성 시각(4,771초)으로 산출했다. 실제 시작은 첫 파일 생성보다 약간 앞서므로 실제 소요는 80분보다 다소 길고, 따라서 실제 배수는 8.2배보다 약간 크다.
- 속도는 12코어 환경 기준이다. 코어 수가 다르면 전체 배수는 달라질 수 있다(사진 1장 단독 6.6배는 환경 영향이 적다).
- 한 장씩 단독 측정값은 처리가 무거운 표본 위주라, 가벼운 사진이 많은 폴더에서는 평균 배수가 더 낮을 수 있다. 단 무거운 사진이 전체 시간을 좌우하므로 전체 배수(8.2배)가 실사용 체감에 가깝다.

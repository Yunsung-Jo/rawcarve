# rawcarve

ddrescue 등으로 복구한 손상된 디스크 이미지(.img)에서
JPEG 이미지와 AVI 영상 파일을 추출하고, 손상된 JPEG를 복구하는 파일 카빙 도구.

- **carve.py** — 디스크 이미지에서 JPEG/AVI를 시그니처 기반으로 추출
- **recover.py** — 추출된 손상 JPEG의 비트스트림 디싱크를 resync 엔진으로 복구

## 작업 규칙

- **main에 직접 커밋 금지.** 커밋 전 현재 브랜치가 main이면 feature 브랜치를 먼저 생성한다.
  (스쿼시 머지의 최종 커밋은 main 위에서 일어나는 *승인된 예외*다 — 규칙은 피처 작업의 직접 커밋을 막는 것이다.)
- 브랜치명: `<타입>/<짧은-설명>` (예: `feat/avi-recovery`, `fix/jpeg-scan`)
- 로컬 머지 시 항상 `--squash` 플래그를 사용한다.
  (`git merge --squash <branch>` → 커밋 → `git branch -d <branch>`)

## 작업 워크플로

- 헤더 복구·색상 복구·이미지 밀림·복구율 개선 등 **정답 원본이 없는 복구/개선 작업**은
  `experiment-loop` 스킬을 따른다 — 문제 정리(사용자와 충분히 소통)·고정 샘플·베이스라인·자율
  루프(가설→실험→예측→육안)·전체 적용(이식→샘플 재검증→전수)·리파인 패스. 스펙이 앞서는 기능
  구현엔 쓰지 않는다.
- off-target 발견(목표와 다른 문제·새 발견)은 **쫓지 않고 라우팅**한다: 포맷 사실→reference,
  off-target 문제→`docs/backlog.md`, 목표 자체를 다시 잡게 만드는 발견→현재 조사 기록 + 승인 후
  목표 재설정.

## 커밋 메시지 규칙

Conventional Commits 형식, 한글. 제목 50자 이내, 본문에 변경 이유와 맥락을 담는다.
`Co-Authored-By` 트레일러 필수.

## 문서

@docs/README.md

- 문서 유형(스펙·ADR·조사 기록·보고서·레퍼런스)을 작성·갱신할 때는 대응하는 작성 스킬을 호출한다 — 목표·독자·문체·제목·자가점검을 따른다: `write-spec` · `write-adr` · `write-investigation` · `write-report` · `write-reference`. (템플릿은 구조를, 스킬은 작성 품질을 담당한다.)
- 문서를 작성·수정한 뒤에는 `review-doc` 스킬로 검토한다 — 수치 원자료 재계산·코드 참조 grep·링크·단정/추론 구분·검증 실행을 능동 확인한다(작성 자가점검이 못 잡는 정밀도 결함을 거른다).
- 비자명한 기술 결정(설계 대안 선택, fallback 전략, 포맷 처리 방식 등) 중 **미래 변경이 위반할 수 있는 지속 제약이나 오래 남는 갈림길**이면 ADR을 작성한다(단순히 병목을 찾아 고친 발견은 조사 기록만으로 충분 — 트리거 기준은 `write-adr`). 결정이 확정되면 상태를 Accepted로, 기존 결정이 대체되면 Superseded로 업데이트한다.
- **리파인 패스 — main에 `--squash` 머지하기 직전에** 아래를 생성·업데이트하고 `review-doc`으로 검토한다.
  루프 중 라이브로 쌓인 조사 기록·레퍼런스를 결론 문서로 증류하는 단계다(백지 작성이 아니다):
  - 영향받은 spec
  - architecture.md
  - README.md
  - 필요한 ADR(결정 관련 결론은 full하게, 과정은 조사 기록 링크)·보고서, 그리고 루프 중 못 남긴 조사 기록·레퍼런스

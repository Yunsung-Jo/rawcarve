# ADR 작성

## 목표

**처음 보는 사람이 3분 안에 "무엇을 결정했고 왜 그렇게 했는지"를 납득하게 만든다.**

구조(절 구성)는 `docs/adr/0000-template.md`를 따른다. 이 스킬은 구조가 아니라
**무엇을 어떤 문체로 쓰는가**를 다룬다. 좋은 ADR과 나쁜 ADR을 가르는 것은 절 제목이
아니라 본문의 초점이다.

## When to Use

- 설계 대안 중 하나를 골랐을 때 (왜 다른 것을 버렸는지가 코드에 안 남는다)
- fallback 전략·포맷 처리 방식·임계값/상수 선택처럼 "왜 이 값/방식인지"가 비자명할 때
- 기존 결정을 대체(Superseded)하거나 확정(Accepted)할 때
- `finishing-a-development-branch` 직전 (CLAUDE.md: 결정이 있으면 ADR을 먼저 쓴다)

## 핵심 원칙

### 1. Why > How

코드를 읽으면 알 수 있는 것(어떻게 구현했는지)은 적게, 코드로는 알 수 없는 것
(왜 이 방식을 택했고, 어떤 대안을 어떤 이유로 버렸는지, 당시 제약은 무엇이었는지)은
많이. **"결정" 절보다 "배경"과 "대안" 절이 더 충실해야 한다.**

### 2. 버린 대안이 본문의 절반이다

"대안" 표의 각 행은 "그 대안을 택했다면 구체적으로 무엇이 나빠지는가"를 적는다.
"별로여서", "복잡해서"는 기각 이유가 아니다. 가능하면 측정/수치로.

- 나쁨: `통째로 추출 | 부정확함`
- 좋음: `통째로 추출 | 상한이 헐거워 잉여 포함. raw를 5→9 MB로 늘리면 gray 0.144→0.517로 악화`

### 3. 결정은 근거로 정당화한다

임계값·상수를 정했다면 "왜 그 값인지"를 분포·실험으로 뒷받침한다.
"0.3으로 정했다"가 아니라 "분포가 겹쳐 깨끗한 경계가 없고, 오판이 무해(비대칭)하므로
낮은 고정값 0.3"처럼. 측정 근거가 없는 결정은 다음 사람이 마음대로 뒤집는다.

### 4. 문체 — 건조하게

감정어·서사("열심히 분석한 결과", "안타깝게도")와 모듈 의인화("정직했다", "삼켰다")를
배제하고 사실과 인과만 남긴다. 전체 규칙·예시는 [공통 문체 규칙](../shared/doc-style.md).

## 제목 규칙

제목은 **결정(상태·결과)**을 명시한다. '규명', '~에서 ~로' 같은 과정의 극적 포장은
검색·스캐닝에 불리하므로 쓰지 않는다. (전체: [공통 문체 규칙](../shared/doc-style.md))

- ✗ `회색 잔존의 주원인 규명: resync 한계 가설에서 carve 가짜 EOI로`
- ✓ `carve의 가짜 EOI 오인을 EOI-직후 엔트로피 검사로 방지` (결정+메커니즘)

## 절차

1. **번호 확정** — `docs/adr/` 최신 번호 +1 (`NNNN`).
2. **템플릿 복사** — `0000-template.md` → `NNNN-짧은-제목.md` (소문자, 하이픈).
3. **본문 작성** — 위 원칙대로. 포맷·스펙 지식은 인라인하지 말고 `docs/reference/`를 링크.
4. **상태 갱신** — 이미 구현·확정된 결정은 `Accepted`, 아직 논의 중이면 `Proposed`로 시작한다.
   이 결정이 기존 ADR을 대체하면 그 ADR의 상태를 `Superseded by NNNN`으로 바꾼다.
5. **목록 갱신** — `docs/adr/README.md`의 Accepted(또는 Superseded) 표에 한 줄 추가.

## 작성 후 자가 점검

- [ ] 처음 보는 사람이 3분 안에 결정과 이유를 말할 수 있나?
- [ ] "왜"가 "어떻게"보다 분량이 많은가?
- [ ] 기각한 각 대안에 구체적(가능하면 정량적) 기각 이유가 있나?
- [ ] 임계값/상수에 "왜 그 값인지" 근거가 있나?
- [ ] 제목만 보고 무엇을 결정했는지 아나? '규명'·'~에서 ~로'가 없나?
- [ ] 모듈을 의인화하거나 감정어를 쓴 문장이 없나?

## Common Mistakes

| 실수 | 수정 |
|------|------|
| 구현 방법을 길게 설명 | How는 코드/스펙에. ADR은 Why에 집중 |
| "대안" 표에 기각 이유가 한 단어 | 그 대안의 구체적 손해를 수치로 |
| 임계값을 근거 없이 단정 | 분포·실험으로 정당화 |
| "정직했지만", "삼켰다" 등 의인화 | 설계된 동작과 인과로 |
| 제목에 '규명', '~에서 ~로' | 결정·결론을 명시한 제목 |
| README 목록 갱신 누락 | 파일 생성 직후 반드시 추가 |

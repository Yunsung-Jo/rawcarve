# 보고서 작성

## 목표

**이 작업이 얼마나 가치 있었는지를 숫자와 데이터로 증명한다.**

이 문서를 읽는 사람은 **개발 백그라운드가 얕은 기획자(PM)·운영팀·상위 관리자**일 확률이
높다. 이들은 코드가 아니라 **숫자와 비즈니스 임팩트**를 본다.

구조(절 구성)는 `docs/reports/0000-template.md`를 따른다. 이 스킬은 그 안을 어떻게
채우는가를 다룬다.

## When to Use

- before/after를 비교해 작업의 효과를 제시할 때 (버전·산출물·표본 대조)
- 배치 처리 결과·평가 결과를 정리해 보고할 때
- 비기술 독자에게 "무엇이 얼마나 좋아졌나"를 전달해야 할 때

과정의 가설·막다른 길은 여기 두지 않는다 — 그건 `investigations/`(랩 노트)의 몫이다.
보고서는 **확정된 결과·수치·시각 증거·권고**만 담는다.

## 핵심 원칙

### 1. 철저한 두괄식 (TL;DR)

맨 위에 **한 줄 요약 + 핵심 결과 표/그림**을 둔다. 바쁜 독자가 거기까지만 읽어도
결론과 임팩트를 알게 한다. 결론을 문서 끝까지 미루지 않는다.

### 2. 정량 데이터 우선

"많이 좋아졌습니다" 대신 숫자를 발췌해 강조한다. before→after, 개수, 비율로.

- ✗ "회색이 크게 줄었습니다."
- ✓ "회색 잔존(gray≥30%) 101개 → 78개. 의심 92개 중 26개 복구, 악화 0."

### 3. 수치 정확성을 반드시 재검토한다 — 보고서 수치는 생각보다 자주 틀린다

**이 단계를 건너뛰지 않는다.** 본문에 적은 모든 핵심 수치를 원자료(`report.csv`,
파일 크기, 로그 등)에서 다시 확인한다.

- **독립 재계산**: 본문 수치를 기억이나 중간 메모가 아니라, 원자료에서 직접 다시 뽑아
  대조한다. 합계·개수·비율·중앙값은 특히 자주 어긋난다. 규모가 크면 별도 스크립트나
  서브에이전트로 핵심 수치를 독립적으로 재산출해 본문과 맞춰 본다.
- **출처 명기**: 각 핵심 수치가 어느 파일·명령에서 나왔는지 부록("사용한 방법·도구")에
  남겨, 독자나 미래의 내가 재검토할 수 있게 한다.
- **단위·기준 일치**: 비교 대상이 같은 표본·같은 기준인지 확인한다(예: 같은 92개 집합,
  같은 budget 조건).

### 4. 각 수치에 의미(임팩트) 한 줄

비전문 독자는 raw number만으로는 "그래서 좋은 건지" 모른다. 각 결과가 무엇을 뜻하는지
한 줄로 해석을 붙인다. ("악화 0 = 정상 파일을 망치지 않는다.")

### 5. 전문용어를 풀어 쓴다 — 독자는 비기술자다

이 보고서는 PM·운영·관리자가 읽는다. 핵심 지표·용어는 **처음 등장할 때 한 줄로 풀어 쓰고**,
본문은 비전문 독자가 따라올 어휘로 쓴다. 코드 식별자·내부 약어를 풀이 없이 쏟지 않는다.

- ✗ "gray≥30% 101→78, hole 발생 92개, resync로 미복원."
- ✓ "회색(데이터가 없어 복원 못 한 영역)이 30% 이상인 파일이 101개 → 78개로 줄었다."

내부 식별자(`FF D9`, MCU, destuff 등)가 꼭 필요하면 괄호로 한 줄 풀이를 단다. 풀이 없이는
쓰지 않는다.

### 6. 한계를 명시한다

표본·측정의 한계, 일반화 시 주의점. 과장 없는 보고가 신뢰를 만든다.

### 7. 문체 — 건조하게, 의인화·문학 표현 금지

- **모듈 의인화 금지**: 모듈은 명시된 동작을 수행하는 시스템이다("삼켰다 / 짜내다 /
  정직했다" 금지 → 설계된 동작과 인과로).
- **감정어·서사 배제**: "극적으로 / 안타깝게도 / 마침내" 등 금지.

비기술 독자가 읽으므로 이런 표현은 특히 오해를 키운다. 전체 규칙·예시는
[공통 문체 규칙](../shared/doc-style.md) 참조.

## 제목 규칙

제목은 **결과(Outcome)**를 담는다. '규명', '~에서 ~로' 같은 과정 포장은 쓰지 않는다.

- ✗ `회색을 줄이기 위한 carve 개선 여정`
- ✓ `carve 가짜 EOI 수정: 의심 92개 중 26개 복구, 악화 0`

## 절차

1. **템플릿 복사** — `0000-template.md` → `YYYY-MM-DD-짧은-제목.md`.
2. **대표 그림 — 작성 전 반드시 질문**(`AskUserQuestion`). 임의로 정하지 않는다:
   - **표본**: 어떤 파일을 넣을지(보통 임팩트가 큰 before→after 1~2개).
   - **블러 여부**: 개인정보 보호를 위해 산출물을 블러할지. 회색/글리치처럼 식별 불가한
     경우는 불필요할 수 있다.
   - **형식은 묻지 않는다** — 레이아웃·폰트·크기는 `tools/montage.py`가 표준으로 처리한다.
3. **대표 그림 생성** — `tools/montage.py`에 스펙(JSON)을 넘겨 만들고 본문 "1. 한눈에 보기"에
   배치한다(아래 "비교 몽타주 규격").
4. **본문 작성** — 두괄식, 정량 우선, 각 수치에 해석.
5. **수치 검증 패스** — 원칙 3대로 모든 핵심 수치를 원자료에서 독립 재계산해 대조.
6. **목록 갱신** — `docs/reports/README.md` 표에 한 줄 추가.

## 대표 그림 — 비교 몽타주 규격

before/after를 한눈에 보여주는 **격자형 몽타주**가 보고서 대표 그림의 표준이다.
레이아웃·폰트·크기 같은 고정 형식은 `tools/montage.py`가 처리하므로 **매번 형식을 정하지
않는다.** 변하는 것(표본·조건·라벨 텍스트·이미지·블러)만 스펙(JSON)으로 넘긴다.
참고 예시: `docs/reports/assets/*-comparison.webp`.

**워크플로**
1. 각 조건의 손상 JPEG를 이미지(PNG 등)로 디코딩한다 — 보고서별 작업(스크립트 밖).
   같은 표본은 모든 조건에서 동일 크롭·배율로.
2. 스펙을 채운다. 행 = 표본 파일, 열 = 비교 조건(좌→우 시간순).
3. `python tools/montage.py <스펙>.json` → `assets/<YYYY-MM-DD>-<주제>-comparison.webp` 생성.

**스펙 형식**
```json
{
  "title": "<작업명> — <맥락(커밋/수정)>",
  "columns": ["기존 (carve 조기종료)", "신규 (EOI 검증 + budget 0)"],
  "rows": [
    {"label": "0x9906A000.jpg", "sub": "회색 0.918 → 0.002", "cells": ["a.png", "b.png"]},
    {"label": "0x96B61000.jpg", "sub": "회색 0.999 → 0.032", "cells": ["c.png", "d.png"]}
  ],
  "blur": 3,
  "out": "docs/reports/assets/2026-06-29-carve-eoi-comparison.webp"
}
```
- `columns` 2~3개, `rows` 개수는 자유(격자는 자동). `cells` 길이는 `columns`와 같아야 한다.
- `sub`(핵심 지표 `before → after`)는 선택. `blur`는 전체 기본 블러 반지름(0=없음, 보통 3);
  행마다 `"blur"`로 덮어쓸 수 있다. **블러 여부는 절차 2에서 사용자에게 묻는다.**
- 고정 스타일(맑은 고딕, 폭 ≈1800px, 흰 배경, WebP)을 바꿀 일이 생기면 상수는
  `tools/montage.py` 상단에 모여 있다.

## 작성 후 자가 점검

- [ ] 대표 그림 표본·블러 여부를 작성 전에 사용자에게 물었고, `tools/montage.py`로 만들었나?
- [ ] 맨 위 한 줄 요약 + 핵심 표/그림만 읽어도 결론·임팩트를 아나?
- [ ] "좋아졌다"류 정성 표현이 모두 수치로 대체됐나?
- [ ] **모든 핵심 수치를 원자료에서 다시 확인했나? 출처가 부록에 있나?**
- [ ] 비교 대상이 같은 표본·기준인가?
- [ ] 각 수치에 비전문 독자용 의미 한 줄이 있나?
- [ ] 핵심 지표·내부 약어를 첫 등장 시 풀어 썼나? 비기술 독자가 읽히나?
- [ ] 한계를 솔직히 적었나?
- [ ] 제목이 과정 포장 없이 결과를 담나? 의인화 표현이 없나?

## Common Mistakes

| 실수 | 수정 |
|------|------|
| 결론을 문서 끝에 배치 | 맨 위 TL;DR로 |
| "많이 개선" 같은 정성 표현 | before→after 수치로 |
| 수치를 기억·중간 메모에서 그대로 옮김 | 원자료에서 독립 재계산해 대조 |
| 다른 표본/기준을 비교 | 같은 표본·조건인지 확인 |
| 수치만 나열, 의미 없음 | 각 수치에 임팩트 한 줄 |
| 코드 식별자·약어를 풀이 없이 나열 | 첫 등장 시 한 줄 풀이, 비기술 어휘로 |
| 과정·막다른 길을 보고서에 | 그건 investigations/로 |
| 수치 출처 미기재 | 부록에 파일·명령 명기 |

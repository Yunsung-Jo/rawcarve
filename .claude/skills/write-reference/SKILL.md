# 레퍼런스 문서 작성

## 목표

외부 포맷·스펙·API 지식, 또는 작업 중 확립한 포맷 사실을 `docs/reference/`에 모아,
다음 작업이 같은 것을 다시 조사하거나 추측하지 않게 한다.

레퍼런스는 **살아있는 문서**다 — 지식이 늘면 갱신·확장한다(스냅샷인 조사 기록과 다르다).

## When to Use

- 외부 스펙·포맷·API가 불확실해 **찾아봐야** 할 때 (마커, RIFF 청크, RFC 조항, 라이브러리 동작)
- 이미 **아는** 포맷 지식을 처음으로 문서화할 때
- 작업 중 **실측·발견한** 포맷 사실을 보존할 때 (비트플립 패턴, 오프셋, 경험 임계)
- 기존 레퍼런스에 새 사실을 **덧붙일** 때

## 출처 모드가 경로를 가른다 (핵심)

레퍼런스 지식은 출처가 셋이고, **출처에 따라 작성 경로가 다르다.** 한 문서가 섞일 수도 있다.

| 출처 | 작성 경로 | 서브에이전트 |
|------|-----------|--------------|
| ① **외부 수집** (모른다 / 권위 필요) | 검색해 확인 후 작성 | **위임한다** — 검색 노이즈를 메인 컨텍스트에서 격리 |
| ② **내가 아는 지식** | 메인에서 직접 작성 + **검증** | 위임 안 함 (검색 불필요) |
| ③ **작업 중 실측** | 메인에서 직접 작성 | **위임 금지** |

**②③을 서브에이전트로 위임하지 마라.** 서브에이전트는 *검색 노이즈를 격리*하는 장치다.
- ③ 실측 지식(방금 잰 오프셋·비율·gray 값)은 **지금 이 작업 컨텍스트에만** 있다. 차갑게
  시작하는 서브에이전트는 그걸 모른다 — 위임하면 지식을 잃는다.
- ② 아는 지식은 검색이 필요 없으니 서브에이전트를 띄울 이유가 없다. **단, 검증에 외부
  검색이 필요하면 그것은 ②가 아니라 ①이다**(위임). ②의 검증은 로컬 코드(`carver/...`)
  대조에 한한다 — 메인 컨텍스트에서 직접 웹 검색해 격리 목적을 깨지 않는다.

## 신뢰 등급 — 출처를 정직하게 표기한다

출처마다 신뢰도와 의무가 다르다. `출처:` 줄에 정직하게 적는다.

- **공식 스펙 / 외부 문서** — 링크 필수. 가능하면 조항(§) 인용. (예: `ITU-T T.81 §F`)
- **실측** — 어떻게·어떤 표본에서 쟀는지. **프로젝트 한정**일 수 있음을 표시(보편 사실 아님).
  한 문서에 스펙과 실측이 섞이면 실측 구간을 `(실측)`으로 표시한다.
- **내가 아는 지식(모델 지식)** — 할루시네이션 위험이 가장 크다. 그대로 적지 말고 **권위 있는
  출처나 코드(`carver/...`)와 대조해 검증한 뒤** 적는다. 검증했으면 출처를 *검증한 그 문서*로
  적는다("모델이 안다"가 아니라). 검증 못 한 항목은 적지 않거나 `(미검증)`으로 표시한다.

## 절차

1. **기존 확인** — `docs/reference/`와 `README.md`를 본다. 같은 주제 문서가 있으면 **새로
   만들지 말고 그 문서에 절을 덧붙인다**(확장). 없으면 새로 만든다.
2. **출처 모드 판단** — 위 표대로 경로를 정한다.
   - ① → "외부 수집 위임"(아래)으로 서브에이전트에 맡긴다.
   - ②③ → 메인 컨텍스트에서 직접 작성한다.
   - **혼합(①+③)** → ① 부분만 위임해 파일을 만들게 하고, 반환 후 메인에서 ③ 실측 절을
     `(실측)` 표시로 덧붙인다(서브에이전트에 ③를 넘기지 않는다). 흔한 경우다(예: `jpeg-markers`).
3. **범위 확정** — 무엇을 담고 무엇은 다른 문서에 두는지 한 문장으로.
4. **작성** — 템플릿(`0000-template.md`)을 복사하거나 기존 문서에 절을 추가. 현재 작업에 필요한
   핵심만(전체 스펙 복사 금지). 관련 코드(`carver/...`)·다른 reference·ADR을 상호링크한다.
5. **검증** — ② 모델 지식은 권위 출처/코드와 대조한다. ③ 실측은 재현 근거(방법·표본)를 남긴다.
6. **목록 갱신** — `docs/reference/README.md` 표에 한 줄 추가(신규 문서일 때).

## 외부 수집 위임 (① 일 때만)

`general-purpose` 서브에이전트에 맡긴다 — 검색 결과가 메인 컨텍스트에 쌓이지 않게 하기 위함이다.
절차 1·3(기존 확인·범위 확정)은 이미 끝낸 상태로 넘기고, 검색 고유 단계만 지시한다:

1. **검색** — WebSearch / WebFetch로 공식 문서·RFC·Wikipedia·소스 리포지터리를 확인한다.
2. **핵심만 추출** — 작업과 직접 관련된 내용만. 전체 스펙을 옮기지 않는다.
3. **파일 생성·목록 갱신** — 템플릿 복사로 `docs/reference/<주제>.md`를 만들고 `README.md`에 추가.
   (기존 문서를 확장하는 경우면 새 파일 대신 그 문서에 절을 추가하게 한다.)
4. **완료 보고** — 생성 경로·담긴 내용·출처 링크를 한 단락으로 요약해 반환한다.

메인 컨텍스트는 요약만 받아 작업을 재개한다.

## 파일명 규칙

`<주제>.md` — 소문자, 하이픈 구분. 예: `jpeg-markers.md`, `avi-riff.md`, `pillow-image-api.md`.

## 작성 후 자가 점검

- [ ] 실측·아는 지식을 메인에서 직접 썼나(③를 서브에이전트로 위임하지 않았나)?
- [ ] 혼합 문서면 ① 부분만 위임하고 ③ 실측은 메인에서 덧붙였나?
- [ ] 모델 지식 항목을 코드/출처로 검증하고 *검증한 출처*로 표기했나?
- [ ] 스펙·실측이 섞이면 실측 구간을 `(실측)`으로 표시했나?
- [ ] 출처 링크와 README 목록(신규 문서)을 채웠나?

## Common Mistakes

| 실수 | 수정 |
|------|------|
| 실측·아는 지식을 서브에이전트로 위임 | 메인에서 직접 작성 (위임은 외부 수집 ①만) |
| 모델 지식을 검증 없이 사실로 기재 | 권위 출처·코드와 대조 후, 검증한 출처로 표기 |
| 스펙과 실측을 구분 없이 섞음 | 실측 구간을 `(실측)`으로 표시 |
| 스펙 전체를 복사 | 현재 작업에 필요한 절만 |
| 같은 주제로 중복 문서 생성 | 기존 문서에 절을 덧붙여 확장 |
| 출처 링크·README 목록 누락 | 작성 직후 반드시 기재·추가 |

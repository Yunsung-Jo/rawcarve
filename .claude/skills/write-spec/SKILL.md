# 스펙 작성

## 목표

**코드를 다 읽지 않고도 이 프로그램이 무엇을·어떻게 동작하는지 파악하게 한다.** 그리고 모듈을
수정·삭제할 때 어떤 프로그램이 영향받는지 짚게 한다.

독자는 둘이다 — (a) 프로그램의 동작을 이해하려는 사람, (b) 모듈을 고치며 영향 범위를 보려는 사람.

구조(절 구성)는 `docs/specs/0000-template.md`를 따른다. 이 스킬은 작성 품질을 다룬다.

## 언제 사용하나

- 새 프로그램(carve.py·recover.py 같은 CLI 도구 등)을 추가했을 때
- 코드 변경으로 프로그램의 **관찰 가능한 동작**(인터페이스·출력·파이프라인·엣지 케이스)이 바뀌어
  기존 spec을 갱신해야 할 때
- **리파인 패스(main `--squash` 머지 직전)**에 영향받은 spec을 갱신할 때

experiment-loop 워크플로에서 spec은 **루프 중이 아니라 작업이 끝난 뒤(전체 적용으로 동작이 출하된 뒤)**
갱신한다 — 동작이 확정돼야 기술할 수 있다.

## 핵심 원칙

### 1. 동작(what)을 기술한다 — 선택 근거(why)는 ADR로

경계 기준: **어떻게 작동하고 무슨 조건에 반응하는가**는 spec이다(동작의 일부). "비트레이트가
평균의 4배를 넘으면 디싱크로 본다", "재개점이 거의 같은 비트면 가짜 복구로 거부한다" 같은
**메커니즘·판별 조건**은 spec에 담는다. 반면 **다른 대안 대비 왜 이 방식인가 / 무엇을 기각했나**
만 ADR로 분리·링크한다(`0002-recover`가 설계 근거로 ADR 0001을 링크하는 식).

복잡한 프로그램은 `복구 원리(알고리즘)` 같은 **동작 원리 절을 적응적으로 추가**해도 된다
— 메커니즘이 무엇에 어떻게 반응하는지를 적는 한. "대안 대비 선택 근거"만 ADR에 남긴다.

### 2. 살아있는 문서 — 코드와 동기화한다

spec은 **현재 동작**을 반영한다(스냅샷이 아니다). 코드가 관찰 동작을 바꾸면 spec을 갱신한다.
**어긋난 spec은 없는 spec보다 위험하다** — 독자가 신뢰하고 틀린다.

### 3. 엣지 케이스와 알려진 한계는 필수

정상 경로만 적지 않는다. 비정상 입력·실패 모드·fallback·**처리하지 못하는 것(알려진 한계)**을
남긴다. 한계를 숨기지 않는다(`0002-recover`의 "물리적 소실은 회색으로 남김", 미해결 항목).
한계는 엣지 케이스 표 안에 둬도, 양이 많으면 별도 절(예: `미해결`)로 빼도 된다.

### 4. 영향 추적 — 사용하는 모듈을 적는다

의존 모듈을 빠짐없이 적어, 모듈을 수정·삭제할 때 영향받는 프로그램을 역추적할 수 있게 한다.
이 섹션이 spec의 운영 가치다.

### 5. 포맷·스펙 지식은 링크한다 (인라인 금지)

전제하는 포맷 지식(마커 표·청크 구조 등)은 `docs/reference/`로 링크한다. spec 본문에 복붙하지
않는다 — 중복은 어긋난다.

### 6. 산문은 건조하게 — 의인화·가치어 금지

개요·파이프라인·동작 원리 등 산문 절에서 모듈을 의인화하거나("은폐한다") 가치어를("정직성")
쓰지 않는다. 사실과 인과로 적는다. 전체 규칙은 [공통 문체 규칙](../shared/doc-style.md) 참조.

## 절차

1. **번호 확정** — `docs/specs/` 최신 번호 +1 (`NNNN`).
2. **템플릿 복사** — `0000-template.md` → `NNNN-프로그램명.md`.
3. **본문 작성** — 동작·계약 기술. 엣지 케이스·알려진 한계 포함. 포맷 지식은 reference,
   설계 이유는 ADR로 링크.
4. **상태** — 구현·확정됐으면 `Accepted`, 아직 논의 중이면 `Draft`. 대체 시 기존 spec을 `Superseded`로.
5. **목록 갱신** — `docs/specs/README.md` 표에 한 줄 추가.

> **갱신일 때:** 코드 변경으로 동작이 바뀐 거라면 새 파일을 만들지 말고 **해당 spec을 갱신**하고
> 날짜를 갱신한다. 새 spec은 새 프로그램일 때만.

## 작성 후 자가 점검

- [ ] 코드를 안 읽은 사람이 인터페이스·출력·동작을 파악할 수 있나?
- [ ] 기술한 동작이 **현재 코드와 일치**하나(살아있는 문서)?
- [ ] 설계 이유(why)를 늘어놓지 않고 ADR로 링크했나?
- [ ] 엣지 케이스와 알려진 한계를 적었나?
- [ ] 사용하는 모듈을 적어 영향 추적이 되나?
- [ ] 포맷 지식을 인라인하지 않고 reference로 링크했나?
- [ ] 산문 절에 의인화·가치어("은폐한다", "정직성")가 없나?

## 자주 하는 실수

| 실수 | 수정 |
|------|------|
| 설계 이유·기각 대안을 spec에 서술 | why는 ADR로 분리·링크 |
| 코드가 바뀌었는데 spec 방치 | 동작 변경 시 spec 갱신 (살아있는 문서) |
| 정상 경로만 기술 | 엣지 케이스·알려진 한계 필수 |
| 포맷 표를 spec에 복붙 | `docs/reference/` 링크 |
| 사용하는 모듈 누락 | 영향 추적 위해 반드시 기재 |
| 동작 변경인데 새 파일 생성 | 기존 프로그램이면 그 spec을 갱신 |
| README 목록 갱신 누락 | 신규 spec 생성 직후 추가 |

# 실험 루프 — 정답 원본이 없는 복구·개선 작업

## 목표

이 작업에는 **정답 원본(원래 정상 이미지)이 없다.** 따라서 한 번에 올바른 복구 방법을
구현하려 하지 않는다. 대신 다음을 반복한다: 고정된 샘플 이미지에서 가설을 하나 세우고,
실험으로 검증하고, 틀린 가설을 제거한다. 살아남은 방법만 전체 데이터에 적용한다.

아래 규칙들은 **검증 없이 성공으로 단정하는 것을 막기 위한 것**이다 — 샘플을 고정하고,
베이스라인을 먼저 저장하고, 예측을 실험 전에 기록하고, 회귀를 감시하고, 종료 조건을 정한다.
규칙을 건너뛰지 않는다.

## 언제 사용하나

다음 작업을 시작할 때 이 스킬을 따른다:

- 헤더 복구, 색상 복구, 이미지 밀림(디싱크) 해결, 복구율 개선
- 그 외 **정답 원본이 없어** 샘플 실험으로 방법을 찾아 전체에 적용하는 모든 복구·개선 작업

다음에는 쓰지 않는다: 동작이 이미 명세로 정해진 기능 구현(스펙이 앞서는 작업).

## 전체 흐름

아래는 단계 요약이다. 각 단계의 구체 지시는 다음 절에 있다. 순서를 지킨다.

1. **문제 정리 + 성공 기준** — 루프 밖, 1회. 사용자와 합의한다.
2. **샘플 고정 + 베이스라인 저장** — 루프 밖, 1회.
3. **루프** — 매 사이클: 원인 분석 → 가설 → 예측 기록 → 실험 → 결과 분석 → 조사 기록 작성. 종료 조건까지 반복.
4. **전체 적용** — 본체 이식 → 샘플 재검증 → 전수 적용. 3단계 모두 수행한다.
5. **리파인 패스** — main에 `--squash` 머지하기 직전. 결론 문서를 작성한다.

## 1단계 — 문제 정리 + 성공 기준 (루프 밖, 1회)

이 단계에서 목표가 틀어지면 이후 모든 작업이 틀린 목표를 향한다. 그러므로 이 단계에서만은
사용자와 통신을 많이 한다. 다음을 모두 한다:

1. 사용자 요구를 **내 말로 다시 풀어 되돌린다.** 예: "즉 헤더의 X가 손상돼 Y가 안 되는 것을,
   Z 기준으로 고치려는 것이 맞나?" 모호한 지점은 **추측해서 진행하지 않고 질문한다.**
2. **성공 기준을 아래 세 가지로 나누어 정의하고 브랜치 작업 노트에 적는다.** 작업마다 세 기준의
   비중이 다르다(헤더 복구는 기준 A 중심, 밀림은 기준 B 중심, 색상은 기준 C 중심).

   - **기준 A — 필수 통과 (자동 판정).** 회귀가 없어야 하고(이전에 정상이던 이미지를 깨뜨리면
     실패), 결과가 구조적으로 타당해야 한다(디코드 성공, 해상도·양자화 테이블·크로마
     서브샘플이 내부적으로 일관). 정답 원본 없이 검사할 수 있는 자기일관성 검사다. **기준 A를
     어기면 다른 기준이 아무리 좋아도 그 방법은 실패로 판정한다.**
   - **기준 B — 진척 지표 (반자동 판정).** 측정 가능한 프록시가 옳은 방향으로 움직였는지 본다
     (회색 % 감소, 디코드된 스캔라인 수 증가, 복구된 바이트 수 증가, 디코드되는 이미지 수
     증가 등). 정답을 몰라도 베이스라인 대비 개선 여부는 측정할 수 있다.
   - **기준 C — 육안 판정 (수동 판정).** 기준 A·B를 통과한 후보들을 눈으로 비교한다. "이 결과가
     정답인가"를 묻지 말고(정답이 없으므로 불가능) **"베이스라인과 비교해 어느 쪽이 더
     자연스러운가"를 묻는다.** 몽타주를 생성해 직접 연다.

3. **세 기준 중 하나만으로 성공을 판정하지 않는다.** 이유: 기준 B(프록시)는 속을 수 있고(예:
   회색 영역을 디코드되는 쓰레기로 채우면 회색 %는 내려가지만 결과는 틀림), 기준 C(육안)는
   언제 멈출지 정하지 못하며, 기준 A(필수)는 진척을 재지 못한다. 세 기준을 함께 본다.
4. **성공 기준은 처음에 완벽하지 않아도 된다.** 잠정으로 시작해 루프 중 다듬는다. 단:
   - 기준을 **조이거나 더 명확하게 하는 변경은 내가 한다.**
   - 기준을 **느슨하게 푸는 변경은 사용자 승인을 받는다.** (결과에 맞춰 기준을 낮추는 자기기만
     방지.)
   - 모든 기준 변경은 조사 기록에 명시한다.

## 2단계 — 샘플 고정 + 베이스라인 저장 (루프 밖, 1회)

매 루프마다 샘플이 바뀌면 결과를 비교할 수 없다. 브랜치당 한 번 고정한다.

1. **샘플을 구성한다.** 샘플에는 두 종류를 반드시 함께 넣는다:
   - **고치려는 실패 케이스** (목표 대상)
   - **회귀 가드** — 이미 정상으로 처리되는 이미지. 변경이 멀쩡한 것을 깨뜨리지 않는지
     감시한다. 실패 케이스만 넣으면 트레이드오프를 놓친다.
2. **내가 샘플을 고를 때는 쉬운 것만 고르지 않는다.** 실패 양상의 다양성을 대표하도록 고른다.
   (샘플 몇 장에만 맞고 전체엔 안 맞는 과적합 방지.) 사용자가 직접 지정해도 된다.
3. **베이스라인을 측정해 저장한다.** 변경 전 샘플의 이미지별 상태(몽타주 + 지표 숫자)를
   scratchpad에 파일로 남긴다. 대화 안에만 두면 컨텍스트 압축으로 사라진다.
4. **재생성이 비싸거나 불가능한 베이스라인 숫자는 즉시 조사 기록에 전사한다.** 예: 변경 전
   실전 처리 시간은 한 번 지나가면 재측정이 비싸다. scratch 파일이 살아남기를 기대하지 않는다.

## 3단계 — 루프 (자율, 사용자 통신 최소)

종료 조건을 만족할 때까지 아래 사이클을 반복한다. 매 사이클마다 다음을 순서대로 한다:

1. **원인 분석.** 깨진 샘플 이미지를 직접 열어 무엇이 어떻게 실패하는지 본다. probe 스크립트로
   비트스트림·바이트 값을 들여다본다. **이 단계에서 carver 본체 코드(`carve.py`·`recover.py`·
   `carver/*`)를 수정하지 않는다.** 분석과 조사용 스크립트만 쓴다.
2. **가설 수립.** 구체적이고 반증 가능하게 쓴다. "헤더의 X가 손상돼 Y가 안 된다. Z를 하면
   복구된다"처럼 무엇을·왜 예상하는지 적는다.
3. **예측을 실험 전에 기록한다.** 어느 이미지가 어느 방향으로 움직일지를 **이미지 이름까지**
   적는다. "좋아질 것" 같은 표현은 쓰지 않는다. 예: "#3·#7은 회색에서 복구로 바뀌고, 회귀 가드
   #1·#2는 바뀌지 않는다." (결과를 본 뒤 예측을 사후에 끼워맞추는 것을 막기 위해 반드시 실험
   전에 기록한다.)
4. **실험.** 가설을 **스크래치 하니스에서 샘플에만** 돌린다(여러 방법을 빠르게 비교할 수 있고
   본체를 흔들지 않는다). 변경이 작고 확신이 서면 본체를 직접 수정해 샘플에만 돌려도 된다.
5. **결과 분석.**
   - **자동 판정은 내가 한다.** 디코드 성공·회색 %·구조 타당성·예측한 이미지가 예측한 방향으로
     움직였는지를 측정해 판정한다. **자동 판정에서 떨어지는 가설은 사용자에게 묻지 않고 내가
     버린다.**
   - **육안 판정도 내가 한다.** 자동 판정을 통과한 후보의 몽타주를 직접 열어 베이스라인과
     비교한다. 판정 결과를 기록할 때 **이미지의 시각적 내용은 묘사하지 않고** 지표·결함 패턴으로만
     적는다.
   - **단, 새 문제에서 처음 그럴듯한 후보가 나오면 그때 한 번 사용자에게 보여 기준을 맞춘다**
     ("내가 복구됨이라고 판정하는 것이 원하는 결과가 맞나"). 이 1회 확인 이후에는 자율로
     판정한다.
6. **조사 기록을 작성한다.** 아래 "문서화 규칙"의 사이클 항목 형식대로 한 항목 추가한다.
   가설·예측은 실험 전에, 증거·판단은 실험 후에 적는다.

**다음 경우에는 혼자 반복하지 말고 작업을 멈추고 사용자에게 먼저 확인한다:**
- 육안으로 어느 후보가 나은지 확신이 서지 않을 때
- 시도할 방향이 여러 갈래로 갈려 무엇을 먼저 할지 판단이 필요할 때
- 여러 사이클을 돌려도 진척 지표가 개선되지 않고 정체할 때

**종료 조건 — 다음 중 하나를 만족하면 루프를 멈춘다:**
- **성공 종료.** 기준 A 유지(회귀 0, 구조 타당) + 기준 B 충족(프록시 목표 도달) + 기준 C 통과
  (육안으로 베이스라인보다 나음).
- **수렴 종료.** 사이클당 개선폭이 들이는 노력 대비 무의미한 수준으로 떨어지면 멈춘다.
  **완벽한 복구가 불가능한 손상은 그대로 둔다** — 데이터의 한계에서 멈추는 것이 정상이며, 없는
  완벽을 강요하지 않는다.

## 4단계 — 전체 적용 (3단계를 모두 순서대로 수행)

샘플에서 검증한 코드와 실제 출하되는 코드가 다르면 샘플 검증이 무의미해진다. 그러므로 한
번에 전수에 적용하지 않고 아래 세 단계를 모두 거친다:

1. **이식.** 검증된 방법을 carver 본체 코드에 구현한다.
2. **샘플 재검증.** 이식한 **실제 본체 코드로 샘플을 다시 돌려** 스크래치 실험 결과와 일치하는지
   확인한다(육안 + 구조 판정). 일치하지 않으면 이식 과정에 문제가 있는 것이므로 전수로 넘어가지
   않는다.
3. **전수 적용.** 재검증이 일치하면 전체 데이터에 적용한다.

**전수 적용에서 실패가 나오면** 그것은 버그가 아니라 "샘플이 전체를 대표하지 못했다"는 신호다.
**새로 드러난 실패 케이스를 고정 샘플에 추가한 뒤 3단계 루프로 돌아간다.**

## 5단계 — 리파인 패스 (main에 `--squash` 머지하기 직전)

루프 중 라이브로 쌓인 조사 기록·레퍼런스를 결론 문서로 정리한다. 백지에서 새로 쓰는 것이
아니라 이미 쌓인 기록을 결론 문서로 압축하는 것이다. 아래 "문서화 규칙"과 CLAUDE.md의 리파인
패스 항목을 따른다. 작성·갱신은 대응 `write-*` 스킬로, 직후 `review-doc` 스킬로 검토한다.

## 문서화 규칙

문서를 언제 쓰는지는 **그 내용이 사라질 증거·사실인지, 루프가 아직 뒤집을 수 있는 결론인지**로
가른다.

| 유형 | 언제 쓰나 | 지시 |
|------|-----------|------|
| **조사 기록** | 루프 중 **매 사이클** | 과정을 라이브로 기록. 상세할수록 좋다. 막다른 길도 남긴다. |
| **레퍼런스** | 포맷 사실을 확립한 **즉시** | 브랜치와 무관한 사실이라 확립 즉시 안정적. |
| **ADR** | 리파인 패스에서 | 결정·제약·상태·향후 고려를 적는다. **결정 관련 결론은 자세히, 단 과정 서사는 베끼지 말고 조사 기록을 링크한다.** 미래 변경이 위반할 수 있는 지속 제약이나 결과가 오래 남는 갈림길이 있을 때만 쓴다. |
| **스펙·보고서** | **모든 작업이 끝난 뒤** | 동작이 출하되고 최종 수치가 확정돼야 의미가 있다. 루프 중에 쓰지 않는다. |

추가 지시:

1. **루프 중에는 결론 문서(ADR·스펙·보고서)를 쓰지 않는다.** 유동적인 결론을 일찍 굳히면 다음
   사이클이 그것을 뒤집게 되고, 써 둔 결론이 판단을 한쪽으로 묶는다. 결정 지점이 보이면 그
   맥락만 조사 기록에 적어 둔다.
2. **조사 기록 사이클 항목 형식** (한 사이클당 한 항목):
   - **가설 / 세운 이유** — 무엇을 의심했고, 직전 증거에서 왜 그 의심이 나왔나 (실험 전)
   - **실험** — 무엇을 어떻게 측정했나 (실험 전)
   - **예측** — 어느 이미지가 어느 방향으로 (실험 전에 기록)
   - **증거** — 나온 결과를 구체 수치·바이트·결함 패턴으로 (실험 후). **몽타주(이미지)의 시각적
     내용은 묘사하지 않는다** — 육안 판정도 지표·결함 패턴(색 캐스트·밀림 행 범위)으로만 적는다
   - **판단** — 가설을 지지/기각했나, 다음은 무엇인가 (실험 후)

   "시도 X, 안 됨"처럼 적지 않는다. "FFD9에서 1바이트 skip → 회색 8%에서 6%로 내려갔으나
   200~340행은 여전히 어긋남 → 하류에 2차 디싱크 의심"처럼 **재사용 가능한 구체로** 적는다.
3. **측정 원장.** 부피가 큰 원시 출력(report.csv, sha 목록, 타이밍 로그, 프로파일 덤프)은
   scratchpad에 둔다. 보고서·ADR의 숫자는 **이 원장에서 그대로 옮겨 적는다(기억이나 재계산으로
   적지 않는다 — 그러면 숫자가 틀어진다).** 원장은 git에 넣지 않으며, 전사한 뒤 폐기해도 된다.
   단 재생성이 비싼 숫자는 즉시 결론 문서로 전사한다.

## off-target 발견 처리

작업 중 목표와 다른 문제나 새 발견이 나오면 **즉흥적으로 쫓지 않는다.** 곁가지를 바로
추적하면 실험 대상이 오염되고 조사 기록에 무관한 내용이 섞인다. **기록은 즉시 하되 해결 착수는
하지 않는다.** 사용자에게 알리고, 미루거나(대부분) 의도적으로 목표를 다시 잡을지 함께 정한다.

발견 종류에 따라 다음 위치에 적는다:

| 발견 종류 | 적는 곳 |
|-----------|---------|
| 새 **포맷 사실** (예: 이 카메라가 여기에도 X를 쓴다) | `reference/` (즉시) |
| 목표와 다른 **문제·버그** (예: AVI에 별도 디싱크가 있다) | `docs/backlog.md` (현재 조사 기록에 섞지 않는다) |
| **현재 목표 자체를 다시 잡게 만드는** 발견 | 현재 조사 기록 + 사용자 승인 후 목표 재설정 |

## 통신 규칙

- **1단계(문제 정리)** — 통신을 많이 한다. 목표·성공 기준·샘플을 사용자와 합의한다.
- **3단계(루프)** — 통신을 최소화하고 자율로 진행한다. 단 처음 그럴듯한 후보로 1회
  캘리브레이션하고, 위 "멈추고 확인" 조건에 해당하면 먼저 확인한다.
- **4단계(전체 적용) 직전** — 1회 통신한다. 사용자 검토를 받은 뒤 전수 적용하고 본체를 커밋한다.

## 자가 점검

- [ ] 1단계에서 목표를 내 말로 되돌리고 모호함을 질문했나? 성공 기준 A·B·C를 적었나?
- [ ] 샘플에 회귀 가드를 포함했나? 베이스라인을 scratchpad에 저장하고, 비싼 숫자는 즉시
      전사했나?
- [ ] 매 사이클에서 예측을 **실험 전에** 이미지 이름까지 기록했나?
- [ ] 자동 판정에서 떨어지는 가설을 내가 버렸나? 육안 판정을 직접 했나? 첫 후보로
      캘리브레이션했나?
- [ ] 조사 기록을 매 사이클 구체 수치로 작성했나? 루프 중 결론 문서(ADR·스펙·보고서)를 쓰지
      않았나?
- [ ] 전체 적용에서 이식 → 샘플 재검증 → 전수 적용 3단계를 모두 거쳤나?
- [ ] off-target 발견을 쫓지 않고 종류별로 라우팅(reference / backlog / 조사 기록)했나?
- [ ] 리파인 패스에서 결론 문서를 정리하고 `review-doc`을 실행했나?

## 자주 하는 실수

| 실수 | 수정 |
|------|------|
| 한 번에 정답 방법을 구현하려 함 | 가설을 하나씩 반증해 제거하는 반복으로 |
| 성공 기준을 "육안으로 좋으면"으로만 둠 | 기준 A(필수)·B(진척)·C(육안) 셋으로 정의 |
| 샘플에 실패 케이스만 넣음 | 회귀 가드(정상 이미지)도 넣음 |
| 결과를 본 뒤 예측을 끼워맞춤 | 예측은 실험 전에 기록 |
| 스크래치 검증 후 본체에 넣고 바로 전수 | 이식 → 샘플 재검증 → 전수, 3단계 모두 |
| 루프 중 ADR·스펙·보고서 작성 | 맥락만 조사 기록에, 결론은 리파인 패스에서 |
| 곁가지 문제를 즉흥적으로 추적 | 기록 후 라우팅하고 미룸 |
| 보고서 숫자를 기억·재계산으로 적음 | 측정 원장에서 그대로 전사 |

# 조사 기록 작성

## 목표

**미래의 나와 동료가 똑같은 삽질을 반복하지 않도록 과정을 자세히 보존한다.**

이 문서를 읽는 사람은 **작업하지 않은 다른 개발자**, 그것도 지금 비슷한 버그를 마주쳐 급하게
검색해 들어온 사람이다. 이들에게는 **논리의 비약이 없어야 하고, 다시 추적할 만큼 충분한 디테일이
있어야 한다.**

구조(절 구성)는 `docs/investigations/0000-template.md`를 따른다. 이 스킬은 그 안을 어떻게 채우는가를
지시한다.

## 언제 사용하나

- `experiment-loop` 워크플로에서 **루프를 도는 동안, 매 사이클마다 한 항목씩 라이브로** 기록할 때.
  (작업이 다 끝난 뒤 한 번에 쓰지 않는다 — 컨텍스트 압축으로 구체 디테일이 사라진다.)
- 비자명한 버그·현상을 추적해 원인을 찾았을 때(혹은 못 찾고 중단했을 때도)
- 확정 결론을 `adr/`·`spec/`·`reference/`에 적기 **전에**, 거기 도달한 과정을 남길 때

조사 기록은 **시점 스냅샷**이다. 루프 중에는 항목을 덧붙여 키우지만, 작업이 끝나면 동결한다. 이후
코드가 바뀌어도 갱신하지 않는다(새 기록을 쓴다).

## 핵심 원칙

### 1. [현상] → [가설] → [예측] → [실험] → [판단] 서사 구조

파편화된 메모를 시간순 또는 논리순으로 엮는다. "서사 구조"는 **조직 방식**을 말하는 것이지 문학적
윤색을 말하는 게 아니다. 흐름은 서사적으로, 문장은 건조하게 쓴다.

### 2. 논리 비약 금지 — 각 단계는 다섯 요소를 갖춘다

가설을 세웠으면 "**왜 그 가설을 세웠는지**"가 직전 단계의 증거에서 따라나와야 한다. 검색해 들어온
개발자가 한 단계도 건너뛰지 않고 따라올 수 있어야 한다.

각 단계(가설 블록)는 다음 다섯 요소를 갖춘다(템플릿 슬롯과 동일):

- **가설 / 세운 이유** — 무엇을 의심했나, 그리고 직전 증거에서 왜 그 의심이 나왔나
- **실험** — 무엇을 어떻게 측정했나 (재현 가능하게)
- **예측** — 실험 결과가 어떻게 나올지를 **실험 전에** 적는다. 어느 파일이 어느 방향으로 움직일지를
  식별자·지표로 명시한다(예: "#3·#7은 gray 0.9→0.1, 회귀 가드 #1·#2는 불변"). 결과를 본 뒤 예측을
  사후에 끼워맞추는 것을 막는다.
- **증거** — 나온 결과를 수치로 (gray 0.728, skew 3.5x, byte% 98 …). 예측과 대조한다.
- **판단** — 증거가 가설을 지지/기각했나, 그래서 다음은 무엇인가

### 3. 자세히 기록한다 — 분량을 아끼지 않는다

이 문서는 **상세할수록 좋다.** 짧게 요약하지 말고, 다시 추적할 수 있을 만큼 구체적으로 적는다.

- **구체 값으로.** "시도했으나 안 됨" ✗ → "FFD9에서 1바이트 skip → gray 0.08→0.06, 그러나 200~340행은
  여전히 어긋남 → 하류에 2차 디싱크 의심" ✓. 실제 본 바이트 값·오프셋·지표를 그대로 남긴다.
  **단 몽타주(이미지)의 시각적 내용은 묘사하지 않는다** — 육안 판정은 지표(gray%)와 결함
  패턴(색 캐스트·밀림 행 범위 등 기술 용어)으로만 적는다(원칙 6).
- **파일별로.** 여러 샘플을 봤으면 어느 파일에서 무엇을 봤는지 파일 단위로 적는다("전반적으로 개선"
  ✗ → 파일별 before→after).
- **재현 가능하게.** 쓴 스크립트·명령·표본을 같이 적어 다음 사람이 그대로 돌릴 수 있게 한다.
- 분량이 길어지는 것은 문제가 아니다. **모호한 것이 더 나쁘다.** 다듬는 것은 리파인 패스에서 결론
  문서로 증류할 때 한다 — 조사 기록 자체는 길어도 된다.

### 4. 기각된 가설을 반드시 남긴다 — 가장 잊기 쉽고 가장 가치 있다

성공한 결과만 남기고 실패한 과정을 지우면, 다음 사람이 같은 막다른 길을 다시 헤맨다. **시도한 것 /
왜 틀렸는지**를 명시한다. "기각된 가설 / 막다른 길" 절은 비워두지 않는다.

### 5. 막다른 길에는 일반화된 교훈을 붙인다

이 조사에만 국한된 사실이 아니라, 다음에 비슷한 함정을 피하게 하는 한 줄을 남긴다.

- ✗ "rate_mult를 2.5로 낮추니 gray 0.999로 붕괴했다."
- ✓ "rate_mult를 2.5 이하로 낮추면 gray 0.999로 붕괴한다. **누적 평균 rate로는 국소 비트레이트를
  분리할 수 없다** — 다음에 평균 기반 임계로 국소 현상을 잡으려 하지 말 것."

### 6. 식별자·지표로만 기술한다 — 이미지 시각 내용 묘사 금지

개인 사진 내용뿐 아니라 **몽타주·복원 이미지의 시각적 내용(장면·인물·실제 색·구도) 일체를 묘사하지
않는다.** 파일 오프셋, gray·skew·byte% 같은 기술 지표와, 색 캐스트·밀림 행 범위 같은 결함 패턴으로만
적는다. ("자세히"는 *기술 지표*를 자세히 적으라는 뜻이지, 이미지를 묘사하라는 뜻이 아니다.) 전체
규칙은 [공통 문체 규칙](../shared/doc-style.md) 참조.

### 7. 문체 — 모듈 의인화·감정어 배제

조사 과정의 1인칭 서술("~로 가정했다", "~로 기울었다")은 허용하되, 아래는 지킨다:

- **모듈 의인화 금지**: carve가 "삼켰다", recover가 "정직했다" 등 금지 → 설계된 동작과 인과로.
- **감정어·서사 배제**: "마침내 / 극적으로 / 안타깝게도" 등 금지.

전체 규칙·예시는 [공통 문체 규칙](../shared/doc-style.md) 참조.

## 제목 규칙

제목은 **결론(상태·결과)**을 담는다. '규명', '~에서 ~로' 같은 극적 포장은 grep·스캐닝에 불리하므로
쓰지 않는다. 본문은 서사적이어도 제목은 결론을 명시한다.

- ✗ `회색 잔존의 주원인 규명: resync 한계 가설에서 carve 가짜 EOI로`
- ✓ `회색 잔존의 주원인은 carve 가짜 EOI 오인(추출 누락)`

## 절차

1. **루프 시작 시 스텁 생성** — `0000-template.md` → `YYYY-MM-DD-짧은-제목.md`. '한 줄' 필드에는 증상
   키워드(오프셋·지표·증상)를 넣는다. 같은 버그로 검색해 들어온 사람이 grep으로 이 기록을 찾는 지점이다.
   (제목·한 줄은 잠정으로 시작해 결론이 잡히면 다듬는다.)
2. **매 사이클 항목 추가** — 루프를 돌 때마다 "조사 과정"에 한 단계를 추가한다. 가설·실험·예측은 실험
   전에, 증거·판단은 실험 후에 적는다(원칙 2·3).
3. **종료 후 정리** — 루프가 끝나면 "기각된 가설 / 막다른 길"과 "결론"을 채운다. 기각된 가설 절은 필수.
4. **결론 연결** — 확정 사실은 `specs/`·`adr/`·`reference/`의 해당 문서로 링크한다(리파인 패스에서).
5. **목록 갱신** — `docs/investigations/README.md` 표에 한 줄 추가.

## 작성 후 자가 점검

- [ ] 작업하지 않은 개발자가 비약 없이 각 단계를 따라올 수 있나?
- [ ] 모든 단계에 [가설/세운 이유 → 실험 → 예측 → 증거 → 판단] 다섯 요소가 있나?
- [ ] 예측을 실험 전에(결과를 보기 전에) 적었나?
- [ ] 구체 값·파일별 관찰·재현 명령으로 자세히 적었나? ("안 됨" 같은 요약이 아니라)
- [ ] 기각된 가설 절이 채워져 있고, 각각 "왜 틀렸는지"가 있나?
- [ ] 막다른 길마다 일반화된 교훈이 한 줄씩 있나?
- [ ] 몽타주·이미지의 시각적 내용을 묘사하지 않고 지표·결함 패턴으로만 적었나?
- [ ] 제목이 과정 포장('규명' 등) 없이 결론을 담나?

## 자주 하는 실수

| 실수 | 수정 |
|------|------|
| 작업 끝나고 한 번에 작성 | 루프 중 매 사이클 라이브로 (압축 전에) |
| 성공 경로만 남기고 실패를 삭제 | 기각된 가설·이유를 반드시 보존 |
| 가설이 어디서 나왔는지 생략(비약) | 직전 증거에 연결 |
| 예측을 결과 본 뒤에 적음 | 예측은 실험 전에 기록 |
| 증거가 "개선됨" 같은 정성 표현 | 수치로 (gray 0.728→0.014), 파일별로 |
| 몽타주에서 본 이미지 내용을 묘사 | 지표·결함 패턴(색 캐스트·밀림 행)으로만 |
| 짧게 요약 | 기술 지표를 다시 추적할 만큼 자세히 (이미지 묘사가 아니라) |
| 막다른 길에 교훈 없음 | 일반화된 한 줄을 붙여 재사용성 확보 |
| 제목에 '규명', '~에서 ~로' | 결론을 담은 제목 |
| 이후 코드 변경에 맞춰 갱신 | 스냅샷이므로 갱신하지 않음(새 기록을 쓴다) |

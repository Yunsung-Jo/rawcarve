# 조사 기록 작성

## 목표

**미래의 나와 동료가 똑같은 삽질을 반복하지 않도록 맥락을 보존한다.**

이 문서를 읽는 사람은 **작업하지 않은 다른 개발자**, 그것도 지금 비슷한 버그를 마주쳐
급하게 검색해 들어온 사람이다. 이들에게는 **논리의 비약이 없어야 한다.**

구조(절 구성)는 `docs/investigations/0000-template.md`를 따른다. 이 스킬은 그 안을
어떻게 채우는가를 다룬다.

## When to Use

- 비자명한 버그·현상을 추적해 원인을 찾았을 때 (혹은 못 찾고 중단했을 때도)
- 여러 가설을 세우고 검증/기각하며 결론에 도달했을 때
- 확정 결론을 `adr/`·`spec/`·`reference/`에 적기 **전에**, 거기 도달한 과정을 남길 때

조사 기록은 **시점 스냅샷**이다. 살아있는 문서가 아니므로 이후 코드가 바뀌어도 갱신하지 않는다.

## 핵심 원칙

### 1. [현상] → [가설] → [검증/실험] → [결론] 서사 구조

파편화된 메모를 시간순 또는 논리순으로 엮는다. "서사 구조"는 **조직 방식**을 말하는 것이지
문학적 윤색을 말하는 게 아니다. 흐름은 서사적으로, 문장은 건조하게.

### 2. 논리 비약 금지 — 각 판단을 직전 증거에 연결한다

가설을 세웠으면 "**왜 그 가설을 세웠는지**"가 직전 단계의 증거에서 따라나와야 한다.
검색해 들어온 개발자가 한 단계도 건너뛰지 않고 따라올 수 있어야 한다.

각 가설 블록은 다음 네 가지를 갖춘다(템플릿 슬롯과 동일):
- **가설 / 세운 이유** — 무엇을 의심했나, 그리고 직전 증거에서 왜 그 의심이 나왔나
- **실험** — 무엇을 어떻게 측정했나 (재현 가능하게)
- **증거** — 나온 결과, 수치로 (gray 0.728, skew 3.5x, byte% 98 …)
- **판단** — 증거가 가설을 지지/기각했나, 그래서 다음은 무엇인가

### 3. 기각된 가설을 반드시 남긴다 — 가장 잊기 쉽고 가장 가치 있다

성공한 결과만 남기고 실패한 과정을 지우면, 다음 사람이 같은 막다른 길을 다시 헤맨다.
**시도한 것 / 왜 틀렸는지**를 명시한다. "기각된 가설 / 막다른 길" 절은 비워두지 않는다.

### 4. 막다른 길에는 일반화된 교훈을 붙인다

이 조사에만 국한된 사실이 아니라, 다음에 비슷한 함정을 피하게 하는 한 줄을 남긴다.

- ✗ "rate_mult를 2.5로 낮추니 gray 0.999로 붕괴했다."
- ✓ "rate_mult를 2.5 이하로 낮추면 gray 0.999로 붕괴한다. **누적 평균 rate로는 국소
  비트레이트를 분리할 수 없다** — 다음에 평균 기반 임계로 국소 현상을 잡으려 하지 말 것."

### 5. 식별자·지표로만 기술한다 (민감 정보 금지)

개인 사진 내용 등은 묘사하지 않는다. 파일 오프셋, gray·skew·byte% 같은 기술 지표로만.

### 6. 문체 — 모듈 의인화·감정어 배제

조사 과정의 1인칭 서술("~로 가정했다", "~로 기울었다")은 허용하되, 아래는 지킨다:

- **모듈 의인화 금지**: carve가 "삼켰다", recover가 "정직했다" 등 금지 → 설계된 동작과 인과로.
- **감정어·서사 배제**: "마침내 / 극적으로 / 안타깝게도" 등 금지.

전체 규칙·예시는 [공통 문체 규칙](../shared/doc-style.md) 참조.

## 제목 규칙

제목은 **결론(상태·결과)**을 담는다. '규명', '~에서 ~로' 같은 극적 포장은 grep·스캐닝에
불리하므로 쓰지 않는다. 본문은 서사적이어도 제목은 결론을 명시한다.

- ✗ `회색 잔존의 주원인 규명: resync 한계 가설에서 carve 가짜 EOI로`
- ✓ `회색 잔존의 주원인은 carve 가짜 EOI 오인(추출 누락)`

## 절차

1. **템플릿 복사** — `0000-template.md` → `YYYY-MM-DD-짧은-제목.md`.
   - '한 줄' 필드에는 증상 키워드(오프셋·지표·증상)를 넣는다. 같은 버그로 검색해 들어온
     사람이 grep으로 이 기록을 찾는 지점이다.
2. **본문 작성** — 위 원칙대로. 가설마다 4요소, 기각된 가설 절은 필수.
3. **결론 연결** — 확정 사실은 `specs/`·`adr/`·`reference/`의 해당 문서로 링크.
4. **목록 갱신** — `docs/investigations/README.md` 표에 한 줄 추가.

## 작성 후 자가 점검

- [ ] 작업하지 않은 개발자가 비약 없이 각 단계를 따라올 수 있나?
- [ ] 모든 가설에 [가설/세운 이유 → 실험 → 증거 → 판단]이 있나?
- [ ] 기각된 가설 절이 채워져 있고, 각각 "왜 틀렸는지"가 있나?
- [ ] 막다른 길마다 일반화된 교훈이 한 줄씩 있나?
- [ ] 민감 정보 없이 식별자·지표로만 기술했나?
- [ ] 제목이 과정 포장('규명' 등) 없이 결론을 담나?

## Common Mistakes

| 실수 | 수정 |
|------|------|
| 성공 경로만 남기고 실패를 삭제 | 기각된 가설·이유를 반드시 보존 |
| 가설이 어디서 나왔는지 생략(비약) | 직전 증거에 연결 |
| 증거가 "개선됨" 같은 정성 표현 | 수치로 (gray 0.728→0.014) |
| 막다른 길에 교훈 없음 | 일반화된 한 줄을 붙여 재사용성 확보 |
| 제목에 '규명', '~에서 ~로' | 결론을 담은 제목 |
| 이후 코드 변경에 맞춰 갱신 | 스냅샷이므로 갱신하지 않음(새 기록을 쓴다) |

# 공통 문체 규칙

문서 작성 스킬(`write-spec`·`write-adr`·`write-investigation`·`write-report`·`write-reference`)이
공유하는 문체·제목 규칙의 정본. 각 스킬은 핵심을 인라인하고 전문은 이 문서를 링크한다.

## 모듈 의인화·문학 표현 금지 (모든 산문 공통)

모듈은 **명시된 동작을 수행하는 시스템**이지 의지·감정을 가진 행위자가 아니다.
"정직했다 / 속았다 / 삼켰다 / 짜내다 / 헤매다 / 은폐한다" 같은 의인화·서사 표현과
"정직성" 같은 가치어를 배제하고, 설계된 동작과 인과로 서술한다.

- ✗ "recover는 없는 데이터를 만들 수 없으니 그 영역을 회색으로 남겼다 — 정직했지만,
  회색의 주원인은 데이터 손실이 아니라 추출 누락이었다."
- ✓ "recover는 물리적 데이터가 없는 영역을 회색으로 렌더링하도록 설계되었다. 따라서 해당
  현상의 근본 원인은 recover의 동작 오류가 아닌, carve 모듈의 조기 종료(추출 누락)에 있다."

## 감정어·서사 배제 (모든 산문 공통)

"열심히 분석한 결과", "안타깝게도", "마침내", "극적으로", "다행히" 같은 표현을 쓰지 않는다.
사실과 인과만 남긴다.

## 제목은 상태(State)와 결과(Outcome)를 묘사한다

> 적용 범위: 결론을 제목으로 갖는 유형(ADR·조사 기록·보고서). 스펙 제목은 프로그램명,
> 레퍼런스 제목은 주제명이라 해당 없음.

'규명', '~에서 ~로' 같은 표현은 과정을 극적으로 포장한 것으로, 검색(grep)·빠른 스캐닝에
불리하다. 제목은 무엇을 결정했는지/무엇이 결론인지를 명시한다.

- ✗ `회색 잔존의 주원인 규명: resync 한계 가설에서 carve 가짜 EOI로`
- ✓ `carve의 가짜 EOI 오인을 EOI-직후 엔트로피 검사로 방지` (결정+메커니즘)

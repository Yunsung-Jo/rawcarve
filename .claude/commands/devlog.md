개발 일지를 작성한다.

## 저장 위치

`docs/devlog/YYYY-MM-DD-<주제>.md` (오늘 날짜 기준)

## 포함해야 할 내용

다음 섹션을 순서대로 작성한다. 각 섹션은 실제 대화 내용과 결정 사항을 기반으로 작성하고, 짧아도 괜찮으니 내용이 없으면 생략한다.

1. **기획 배경** — 이 작업을 시작한 동기, 해결하려는 문제
2. **주요 결정** — 설계/구현 과정에서 내린 결정들. 선택지가 있었다면 검토한 옵션과 선택 이유 포함
3. **사용자 피드백 / 주요 프롬프트** — 방향을 바꾼 피드백이나 중요한 지시는 원문 그대로 인용 (`>` blockquote 사용)
4. **구현 중 발견된 문제와 수정** — 버그, 설계 변경, 예상 못한 복잡성
5. **최종 결과** — 완료된 것, 동작 확인 방법
6. **알려진 한계** — 미해결 문제, 향후 개선 여지

## 작성 지침

- 서술 형식(narrative)으로 작성. 나중에 "왜 이렇게 됐지?"를 찾아볼 때 맥락을 이해할 수 있어야 한다.
- 기술 용어는 그대로 쓰되, 결정의 이유를 반드시 적는다.
- 코드 변경 내용보다 **결정의 맥락**이 중요하다. 코드는 git으로 확인할 수 있지만 맥락은 일지에만 있다.
- 사용자가 직접 말한 내용(피드백, 지시, 질문)은 원문 그대로 인용한다.

## 완료 후

파일을 저장하고 사용자에게 경로를 알려준다. 커밋 여부는 사용자에게 묻는다.
